## 1. Philosophy

Strategy Factory is not a trading bot. It is a **factory that builds, evaluates, and manages trading bots** — and, over time, learns how to build them better. Every architectural decision flows from six non-negotiable commitments.

### 1.1 Determinism over cleverness

Anything that reads market data, evaluates a strategy, allocates capital, or scores a decision is a **pure function of its inputs**. Given identical inputs, identical outputs. This is why we forbid `datetime.now()` inside evaluators, forbid random shuffles in ranking, and forbid opaque ML models in Phase I's meta-learner.

*Why it matters:* Non-determinism kills reproducibility, and reproducibility is the only way to defend a decision six months later during a post-mortem, an audit, or a regulator's inquiry.

### 1.2 Explainability over performance

Every autonomous decision — a strategy activation, a capital reallocation, a broker health downgrade, a meta-learning weight recommendation — writes a row to `outcome_events` with the exact metrics, evidence IDs, and reasoning that produced it. If it isn't explainable, it doesn't ship.

*Why it matters:* A trading system that cannot answer *"why did you do that?"* is unsafe. Full stop.

### 1.3 Additivity over refactoring

Every new phase adds a package, a router, and (optionally) collections and orchestrator tasks. **No existing file's signature changes.** No engine's behaviour changes unless an operator explicitly opts in via env var.

*Why it matters:* Additivity means a broken Phase J cannot corrupt Phase C. It means feature flags are free (they never mattered because the old code path is still there). It means we sleep at night.

### 1.4 Independence over coupling

Every engine is a **standalone module** with its own types, config, ledger, and API. They talk through *events* (`outcome_events`) and *contracts* (protocols), never through direct state manipulation.

*Why it matters:* The moment two engines share mutable state, one of them will corrupt the other. Independence forces every cross-engine effect to be observable, testable, and reversible.

### 1.5 Bounded blast radius over automation

Autonomous behaviour is bounded — max deltas per tick, class caps per day, kill-switches per broker, mode gates per engine. A runaway loop can only make small mistakes, and every mistake is logged.

*Why it matters:* Speed multiplied by autonomy equals catastrophe. Slow-then-fast beats fast-then-broken.

### 1.6 Backward compatibility over elegance

We would rather ship an ugly `if brain_uses_meta_overrides: ...` branch than break Phase F. Every default is chosen so that a fresh boot produces **byte-identical behaviour to the previous release**.

*Why it matters:* Users trust the system because it doesn't surprise them. Elegance is a nice-to-have; predictability is a requirement.

**Why FastAPI + async Motor?** Non-blocking I/O so hundreds of concurrent learning cycles + orchestrator ticks + API requests share one process. Uvicorn workers are stateless and horizontally scalable, but for now single-process is sufficient (Phase B.1 audit confirmed).

**Why MongoDB and not Postgres?** Every collection is a flat, append-heavy time-series (outcome_events, execution_journal, structural_changes). Document schemas evolve *additively* every phase — moving to a rigid relational schema would gate every deployment on a migration. MongoDB's flexible schema is a feature, not a bug, for a research platform.

**Why no message broker (Kafka/Redis)?** `outcome_events` IS our message bus. Every engine reads what it needs; nothing is broadcast. Adding a broker doubles moving parts and hides state in an ephemeral place. We picked observability over throughput.

## 3. The Nine Engines and Why They're Separate

| # | Engine | Role | Owns collections |
|---|--------|------|------------------|
| 1 | `learning` (Phase A/B/B.1) | Foundational outcome-event emitter + supervisor + continuous scheduler | `outcome_events` |
| 2 | `orchestrator` (Phase B.2) | Central priority-scored task dispatcher | (none; controls other engines) |
| 3 | `intelligence` (Phase C) | Strategy classification, regime detection, activation scoring | (writes to `outcome_events`) |
| 4 | `portfolio` (Phase D) | Adaptive Master Bot portfolio construction, allocation, promotion, retirement | (writes to `outcome_events`) |
| 5 | `brain` (Phase F) | Adaptive Trading Brain — per-strategy scoring, transition pre-staging | (writes to `outcome_events`) |
| 6 | `market_intel_engine` (Phase G) | 8 market observers + change detection + intelligence aggregator | `market_snapshots`, `market_states`, `structural_changes`, `market_intelligence` |
| 7 | `execution` (Phase H) | Broker adapters, order/position lifecycle, ledger, replay, risk monitor | `order_requests`, `fill_events`, `positions`, `broker_health`, `execution_quality`, `execution_attribution`, `execution_journal` |
| 8 | `meta_learning` (Phase I) | Meta-parameter recommendation engine | `meta_learning_evaluations`, `_recommendations`, `_applications`, `_overrides`, `_mode_history` |
| 9 | `ai_workforce` (Phase B) | Provider health + routing telemetry | (writes to `outcome_events`) |

Each engine ships with:
- `types.py` (dataclasses)
- `config.py` (env-driven, live-reload)
- Its own `ledger.py` for owned collections
- An API router under `/api/<engine>/*`
- A pytest module `test_v1_2_0_alpha2_phase_<letter>.py`
- (Optional) one or more orchestrator task adapters

### Why is Trading Brain separated from Execution Intelligence?

They live in different mental universes:
- **Brain** answers *"should we trade this strategy right now, and how much?"* It is a **pricing engine** for portfolio actions.
- **Execution** answers *"if the brain says yes, what actually happened at the broker?"* It is a **measurement engine** for real-world friction.

Coupling them would cause two anti-patterns:
1. **Brain latency inflated by broker calls.** A single broker heartbeat delay would freeze every strategy score.
2. **Execution decisions contaminated by brain state.** If execution could reach into brain scores, a failed fill could accidentally reweight strategies without going through the closed-learning loop — invisible to `outcome_events`.

Instead they communicate through the **execution_attribution** table: brain writes `brain_decision_id` on every order; execution writes back `execution_realised` outcome events joined by that same ID. The join is data, not a function call. It is queryable, testable, and time-travellable.

### Why is Market Intelligence separated from Trading Brain?

Same reason, one level up. Market Intelligence answers *"what regime is the market in? what has structurally changed?"* It reads *only* market data and outcome_events; it never reads strategy state. Trading Brain **consumes** market intelligence via three optional weights (`BRAIN_W_MARKET_CONFIDENCE`, `_STYLE_CONFIDENCE`, `_OPPORTUNITY`), each defaulting to `0.0` — a **two-step opt-in** (Q4 of Phase G design).

If Market Intelligence were embedded in the Brain:
- Every brain tick would recompute market state (wasted CPU) OR the brain would carry stale market state (correctness bug).
- Market observers couldn't be swapped/removed without touching brain code.
- A market observer bug could silently corrupt strategy scores.

By putting Market Intelligence upstream and gating its influence with three-value weights, the operator has a **dimmer switch**, not an on/off toggle.

## 4. The Explainability Chain

Every decision in the system emits **at least one row** to `outcome_events`. The row carries:
- Immutable IDs joining it to upstream causes (`brain_decision_id`, `learning_run_id`, `strategy_hash`, `attribution_id`, `evaluation_id`, `recommendation_id`).
- Metrics that drove the decision.
- Evidence (references to input outcome_events, sample IDs, method markers).
- A human-readable `reason` string.

### The complete chain for a Tier 1 strategy activation:

```
learning:cycle_started
   ↓
learning:strategy_generated
   ↓
learning:backtest_passed
   ↓
intelligence:classify (style + regime_suitability)
   ↓
intelligence:portfolio_scored
   ↓
intelligence:bundle_built (Tier 1 slot 3)
   ↓
portfolio:activate (weight 0.045)
   ↓
brain_tick (regime=trending, market_confidence=0.72)
   ↓
brain_decision (score_now=0.81, action=TRADE_NOW)
   ↓
[execution engine] order submitted → broker_order_id
   ↓
fill_event (with slippage + latency)
   ↓
execution_journal (append-only)
   ↓
position closed
   ↓
execution_attribution (joins brain_decision → realised_pnl)
   ↓
execution_realised (delta_predicted_realised = -0.12)
   ↓
[meta_learning cycle]
   ↓
meta_learning_evaluation (weight_sensitivity computed)
   ↓
meta_learning_recommendation (Δ BRAIN_W_REGIME_FIT = +0.008)
   ↓
[in OBSERVE — stops here]
[in RECOMMEND — awaits operator approve]
[in AUTONOMOUS — meta_learning_application written]
```

**Every arrow in that diagram is a Mongo row you can `find()` six months from now.** That is the explainability chain.

### Why not structured logs?

Logs are ephemeral, hard to query, and coupled to log-shipping infrastructure. `outcome_events` is a first-class data collection: indexed, TTL'd, aggregable, queryable from the same API surface the frontend uses. Logs and outcome_events serve different purposes — logs for latency-sensitive debugging, outcome_events for audit and learning.

## 5. The Unified Orchestrator

Before Phase B.2, every long-running job (data topup, learning cycles, ranking, master bot refresh) had its own APScheduler. That produced:
- Redundant capacity probes (each scheduler polled the host independently).
- No global budget awareness — a runaway LLM job could exhaust the daily budget while a cheap ranking job sat idle.
- No cross-task priority — a housekeeping task and a high-value learning cycle competed for the same CPU with no arbitration.
- Multi-scheduler startup ordering bugs.

The **Unified Orchestrator** replaces all of them with **one priority-scored dispatcher** ticking every N ms, evaluating every registered task's readiness, and dispatching the highest-scoring subset that fits the current CPU/RAM/AI-budget budget.

### The priority formula

```
priority = priority_base
         × business_value
         × freshness_pressure
         × dependency_readiness
         × budget_headroom
         ÷ resource_cost_factor
```

Every task registers its own `priority_base`, `business_value`, `EXPECTED_DURATION_S`, `AI_PROVIDER_REQUIRED`, `COST_ESTIMATE_USD`, and dependencies. The scorer is deterministic — same inputs, same output.

### Why is validation "passive by default"?

Some tasks (validation, optimization, `master_bot_bundle_refresh`, `self_rebuild`) are registered but marked `PASSIVE=True`. The scheduler will *never* dispatch them unless an operator explicitly enables them via `ORCH_TASK_<NAME>_PASSIVE=false`.

That is our "quiet-by-default" posture: new autonomous behaviour is registered so its metadata is inspectable, but only ever runs when an operator opts in. This is a load-bearing safety principle: **the presence of a task is not the same as the activation of a task.**

## 6. The Continuous Scheduler

Phase B.1 replaced APScheduler's fixed-interval learning dispatcher with a **capacity-aware continuous scheduler**. Instead of "run every 60 seconds", the new scheduler:

1. Polls `AdaptiveConcurrency.recommend()` (built on live host CPU/RAM + queue pressure + compute probe).
2. If under the recommended concurrency, dispatches a new learning cycle as an `asyncio.Task`.
3. Otherwise sleeps a short poll interval and retries.
4. Respects a rolling-hour governor to prevent burst overspend.

### Why capacity, not time?

Time-based schedulers waste headroom. On a 16-core box, an APScheduler set to "one cycle per minute" leaves 90%+ of the machine idle. A capacity-based scheduler asks *"can I fit one more?"* and — if the box says yes — dispatches immediately.

The Phase B.1 audit measured **13.7 cycles/second on a fully-loaded 16-core box at 9% CPU utilisation** with `USE_PROCESS_POOL=false`. Root cause was documented in `audit/PHASE_B1_PERFORMANCE_AUDIT.md`: fixed scheduler + serialised `_SIG_LOCK` + serialised Mongo emit.

## 7. The Trading Brain

Phase F introduced the Adaptive Trading Brain as a **continuous action space**: every strategy is scored on `[0..1]` and translated into a target weight, bounded by a per-tick maximum delta.

### Why continuous weights and not INCREASE/DECREASE?

Phase D had discrete actions: `ACTIVATE`, `PAUSE`, `REDUCE`, `INCREASE`, `REPLACE`, `HOLD`. In practice:
- `INCREASE by 10%` and `INCREASE by 30%` were the same action with wildly different consequences.
- Small confidence changes could not express themselves — every strategy either kept the same weight or jumped by the full delta.
- Regime transitions produced whipsawing: pause → activate → reduce → activate as regime probabilities crossed thresholds.

Continuous scoring smooths all of that:
- `score_now = 0.72` maps to `target_weight = 0.036` (for a bundle of 20).
- `score_now = 0.68` maps to `target_weight = 0.034`.
- The tick delta is `−0.002`, bounded by `BRAIN_MAX_WEIGHT_DELTA_PER_TICK=0.05`.

Gradual evolution beats binary choice.

### Why "score_now" and "score_next"?

Phase F added a **regime transition predictor** (`regime_transition.py`). When the transition probability exceeds `BRAIN_TRANSITION_PROB_MIN` (default 0.5), the brain computes `score_next` — the strategy score assuming the market has already transitioned to the predicted next regime. Strategies with high `score_next` and paused status get a **shadow allocation** (`BRAIN_PRE_STAGE_SHADOW=0.03`) — a rehearsal, not real capital.

This is *pre-staging*, and it exists because you don't want to be caught flat-footed when the market regime changes at 2:00 AM UTC and no operator is watching. Pre-staging is a bounded rehearsal — never real capital until fully promoted through Phase D's pipeline.

### Why the catastrophic override?

Every tick, before scoring, the brain checks four kill-switch conditions:
- Strategy drawdown ≥ `BRAIN_EMERGENCY_DD_PCT` (default 30%)
- Confidence ≤ `BRAIN_EMERGENCY_CONFIDENCE` (default 0.15)
- Prediction accuracy ≤ `BRAIN_EMERGENCY_PREDICTION_ACCURACY` (default 0.20)
- Corrupted strategy row or broker failure

Any hit → `EMERGENCY_ZERO`: the strategy's target weight goes to `0.0` this tick, bypassing the gradual-delta gate.

*Why bypass?* Gradual evolution is for *known-good* strategies. When something goes catastrophically wrong, you don't want to spend 6 ticks unwinding — you want to be flat immediately, then investigate.

## 8. Market Intelligence

Phase G's 8 observers watch the market itself:
1. Trend duration
2. Volatility dynamics
3. Breakout quality
4. Reversal strength
5. Session statistics
6. Liquidity estimator
7. Correlation matrix
8. Style performance

Each observer is a **pure function** of market snapshots. Zero Mongo access, zero shared state. This is what makes them unit-testable in isolation.

### Why heuristic change detection instead of ML?

Change detectors are labelled `method="heuristic_*"` (CUSUM-lite volatility, running-mean drift, correlation breakdown). This is intentional:
- Heuristics are **interpretable**: you can hand-inspect the trigger.
- Heuristics are **fast**: no model reload, no batch inference.
- Heuristics are **safe first**: false positives are cheap (a false structural-change signal doesn't unwind the portfolio in Phase G — it only nudges brain weights *if* enabled).
- Once Tier 5 validation has been recorded for many months, Phase G.2 can replace these with ML models that must first *match* the heuristic's outputs on the recorded data — a form of "shadow deployment" for the model itself.

### Why "influence" instead of "instruct"?

Market Intelligence adds three signals to the brain (`market_confidence`, `style_confidence`, `opportunity_score`) with configurable weights. It never directly modifies strategy weights, never triggers a rebuild, never forces a pause.

The only exception is the **market-driven force-pause hook**, which is `OFF by default` (Q5 of Phase G design). Even with the master switch on, that specific behaviour requires explicit operator opt-in via `BRAIN_MARKET_RISK_PAUSE_ENABLED=true`.

*Why?* Because a bug in Market Intelligence could otherwise cascade into portfolio-wide pauses. By defaulting the influence to zero and the pause to disabled, we contain any Phase G defect to the observability plane.

## 9. Execution Intelligence

Phase H is the most safety-critical engine: it is the only one that touches real capital. Its design principles are stricter than any prior phase.

### The Broker Adapter Protocol

```python
class BrokerAdapter(Protocol):
    async def connect(self): ...
    async def disconnect(self): ...
    async def submit(request: OrderRequest) -> Tuple[str, str]: ...
    async def cancel(request_id: str, reason: str) -> bool: ...
    async def health(self) -> BrokerHealth: ...
```

Two adapters ship in Phase H:
- **PaperBrokerAdapter** (default) — deterministic, configurable slippage/latency/reject rates, no external calls.
- **CtraderBrokerAdapter** (opt-in) — wraps a `CtraderTransport` (currently mocked; real Protobuf wiring deferred to H4.1). Resilience layer: `OAuthSession`, `CircuitBreaker` (5 fails/60s → OPEN 5min), `ExponentialBackoff`, `ResilientConnection` (heartbeat every 15s, 3 miss → reconnect).

### Why paper by default?

A fresh boot with no `BROKER=` env goes to paper. To hit a real venue, an operator must consciously set `BROKER=ctrader` **and** provide credentials. This is Q1 of the Phase H design: **default broker is paper; live venue is always an explicit operator flip**.

### Why immutable-ID audit chain?

Every state change carries three IDs:
- `request_id` — client-side idempotency (uuid4, generated before submit)
- `broker_order_id` — assigned by broker
- `brain_decision_id` — links back to the brain outcome_event

All three are journaled in `execution_journal` and appear in `outcome_events`. This is Q5 of Phase H: **the audit chain must be resolvable from IDs alone, without any join tables outside outcome_events + execution_journal**.

### Why is risk monitor recommend-only?

Phase H's `risk_monitor.evaluate_guards()` returns a list of `RiskRecommendation` objects (`pause` / `reduce` / `halt_new_opens`). **It never auto-flattens positions.** This is Q3 of Phase H's design.

*Why?* Auto-flatten is a nuclear option. In a fast market it can lock in losses that would have recovered in the next tick. It can also cascade: a spurious risk signal → auto-flatten → suddenly large loss appears in the ledger → daily-loss guard trips → cascade continues. Recommend-only halts the loop at the first step and requires operator judgement.

### Why the immutable journal?

`execution_journal` is append-only, monotonic-seq per account, with an asyncio.Lock guarding concurrent writers. This is the substrate for **Replay** (§11).

## 10. The LedgerBackend Abstraction

Phase H introduced a `LedgerBackend` Protocol with three implementations:
- **MongoLedgerBackend** (default) — writes to Mongo.
- **MemoryLedgerBackend** — writes to an in-process dict, ~10× faster.
- **ReplayLedgerBackend** (§11) — feeds a deterministic replay.

Selection via `EXEC_LEDGER_BACKEND=memory|mongo` env or explicit `set_backend()`.

### Why the abstraction?

Three independent needs:
1. **CI speed.** Every test that touches the execution engine used to open a Mongo connection. Memory backend cuts tier-1 CI from ~15s to ~2s.
2. **Deterministic replay.** Replay needs to intercept every ledger write to reconstruct terminal state — a real Mongo can't do that from an offline journal.
3. **Zero-durability workloads.** Backtests running 10^6 fills to disk are wasteful when the fills won't ever be re-read.

*Why not just skip persistence in tests?* Because "just skip" is a bug factory. Tests that write to a memory backend catch the same idempotency bugs, seq-collision bugs, and index-order bugs as tests that write to Mongo — they just do it faster. Both backends satisfy the same Protocol.

## 11. Replay

`replay_range(account_id, start_seq, end_seq)` reads the immutable journal and reconstructs terminal state per `request_id` purely from journaled events. The reconstructed state is byte-identical to the live-run state.

### Why determinism as a first-class concern?

Two use cases drive replay:
1. **Debugging.** "The system executed 12,000 orders overnight. Position 4783 shows −$180 realised. Why?" Replay lets you walk from the first journal event of that request to the final fill, seeing every state transition.
2. **Regression testing.** A recorded journal from a real production hour becomes a permanent test fixture. Any future code change that alters terminal state on that journal fails the regression.

*Why append-only?* Because a mutable journal is a rewritable history. History with delete permission cannot be trusted. Append-only guarantees the state you saw in the audit six months ago is the same state you see today.

### Why per-account monotonic seq?

Global seq across accounts would either serialise all writes (kills concurrency) or leave gaps (breaks replay determinism). Per-account seq scales horizontally: 10 accounts can write concurrently without contention, and each account's journal is a totally-ordered stream.

An asyncio.Lock per account guards the seq allocation. When `asyncio.gather()` fires 20 concurrent writers, they serialise on the lock but the total wall-time is bounded by 20 × single-write-latency (Mongo write is ~1ms; total ~20ms).

## 12. Meta-Learning — Why It Starts in Observe Mode

Phase I ships in OBSERVE mode. It reads outcome_events, computes evaluations, ranks recommendations, and **applies nothing**.

### The four operating modes

| Mode | Reads | Writes | Applies to overrides | Applies to live trading |
|------|-------|--------|----------------------|-------------------------|
| Disabled | ✗ | ✗ | ✗ | ✗ |
| **Observe (default)** | ✓ | Recs + Evals only | ✗ | ✗ |
| Recommend | ✓ | Recs + Evals + Apps (operator-approved) | ✓ (per approval) | ✓ (only if `*_USE_META_OVERRIDES=true`) |
| Autonomous | ✓ | Recs + Evals + Apps (auto) | ✓ (whitelisted) | ✓ (only if `*_USE_META_OVERRIDES=true`) |

The last column matters: **even in Autonomous mode, meta-learning writes to `meta_learning_overrides`**, and downstream engines only consume overrides when their own env flag says so. Two safety nets in series.

### Why start in OBSERVE?

Three reasons, in ascending order of importance:

1. **Data warmup.** A meta-learner with 3 data points recommends nonsense. OBSERVE mode collects data until sample sizes meet `META_LEARNING_MIN_SAMPLES` (default 50, with a warmup ramp to 30).

2. **Calibration.** The evaluators (weight_sensitivity, threshold_calibration, etc.) themselves might be miscalibrated. Running them in OBSERVE for several weeks lets us verify that (a) they emit sensible recommendations, (b) the recommendations converge instead of oscillating, (c) their significance scores align with intuition.

3. **Trust.** An operator cannot trust an autonomous meta-learner until they have watched it propose recommendations for real trades and validated that the proposals were the ones they would have made themselves. OBSERVE mode is the co-pilot phase; Recommend mode is the mentor phase; Autonomous mode is the delegated-authority phase. Skipping steps in that ladder produces disasters.

### Why the deterministic heuristics instead of ML?

Same reasons Market Intelligence uses heuristics:
- **Interpretability**: every recommendation carries a Pearson correlation, sample size, and evidence IDs. An ML model would produce a scalar output with no direct trace.
- **Replay-safety**: given identical outcome_events, the meta-learner produces byte-identical recommendation IDs. This is essential for regression tests and post-mortem audits.
- **Bounded surprise**: heuristics can produce boneheaded recommendations, but never *novel* ones. That is a feature — we want the operator to be surprised by nothing.

Phase I.2 (later, not now) may add ridge/logistic regressions where sample sizes justify them, with the heuristic always kept as a floor.

### Why the structural non-modification guarantee?

The design document promised (§17): **no edits to `brain/scorer.py`, `brain/policy.py`, `brain/brain.py`, `brain/config.py`, `portfolio/*`, `execution/risk_monitor.py`, `execution/quality.py`, `execution/attribution.py`, `market_intel_engine/*`.** A pytest hashes those files and fails if any change slipped in.

This is the strongest possible guarantee of additivity: even if Phase I's code has a bug, none of the other engines' behaviour can change unless an operator opts in via `*_USE_META_OVERRIDES=true`. That flag is off by default and never gets set except by conscious operator action.

## 13. The Frontend Decoupling

We deliberately have not shipped a frontend for v1.2.0-alpha2. Backend autonomy is the sole focus.

### Why?

Three reasons.

1. **A frontend written against an unstable backend is a rewrite waiting to happen.** Every phase adds APIs (`/api/orchestrator`, `/api/intelligence`, `/api/portfolio`, `/api/brain`, `/api/market-intelligence`, `/api/execution`, `/api/meta-learning`). Building UI on top of a moving target means the UI is always chasing the backend — never producing decision quality.

2. **Backend correctness is expensive to visualize wrong.** A "green" dashboard on a broken backend is worse than no dashboard at all. Backend validation must complete (Tier 5 24h + 72h) before we invest in visualisation.

3. **The UI/UX Master Design Specification exists.** `docs/UI_UX_MASTER_DESIGN_SPECIFICATION_v1.0.md` (700+ lines) is the canonical frontend constitution. It captures every module, layout, motion pattern, and design token *in text*, so when frontend implementation begins it can hit the ground running.

The frontend will be built once backend validation is signed off — not before, not simultaneously.

## 15. Configuration Surface

Every engine reads env at call-time, not import-time. Flipping a switch changes behaviour on the next tick, never requires a restart (except for structural changes like enabling a new orchestrator task).

Key env categories:
- **Master switches**: `EXEC_ENABLED`, `MI_ENABLED`, `ORCHESTRATOR_ENABLED`, `LEARNING_CONTINUOUS_MODE`, `META_LEARNING_MODE`.
- **Opt-in bridges**: `BRAIN_USES_MARKET_INTELLIGENCE`, `BRAIN_USES_LIVE_EXECUTION`, `BRAIN_USE_META_OVERRIDES`, `PORTFOLIO_USE_META_OVERRIDES`, `EXEC_USE_META_OVERRIDES`.
- **Weights + thresholds**: every scoring weight (`BRAIN_W_*`), threshold (`BRAIN_TRADE_NOW_THRESHOLD`, etc.), and cap.
- **Budgets**: `ORCH_BUDGET_DAILY_USD_GLOBAL`, `ORCH_BUDGET_RPM_<PROVIDER>`.
- **Broker credentials**: `CTRADER_CLIENT_ID`, `CTRADER_CLIENT_SECRET`, `CTRADER_ACCOUNT_ID`.

**No secrets in code. No secrets in config files. All secrets in env.** The `.env` file is `.gitignore`'d.

## 16. Feature Flags and Rollback

Every phase ships behind an off-by-default flag or a mode gate. Rollback is always one env change away.

| Phase | Flag | Rollback effect |
|-------|------|-----------------|
| B.1 | `LEARNING_CONTINUOUS_MODE=false` | Legacy fixed-interval scheduler resumes |
| B.2 | `ORCHESTRATOR_ENABLED=false` | Legacy per-purpose schedulers resume |
| F | `PORTFOLIO_POLICY=phase_d` | Phase D discrete allocation resumes |
| G | `MI_ENABLED=false` | Market observers dormant; brain unchanged |
| G+ | `BRAIN_USES_MARKET_INTELLIGENCE=false` | Brain ignores market signals (byte-identical to Phase F) |
| H | `EXEC_ENABLED=false` | Execution engine dormant |
| H | `BROKER=paper` | Live venue disabled |
| I | `META_LEARNING_MODE=disabled` | Meta-learning task never runs |
| I | `BRAIN_USE_META_OVERRIDES=false` | Overrides ignored by brain (default) |

**Rollback is not "delete the code". Rollback is "flip the flag".** The old code path is always still there.

## 17. Deployment Architecture

Local: `uvicorn` under `supervisord`. React frontend under a separate `supervisord` program (currently deferred).

VPS: Docker Compose overlay (`infra/compose/docker-compose.prod.yml`) with Traefik reverse proxy.

Health probes: `/api/health`, `/api/deployment/status`, per-engine `/api/<engine>/config` and `/api/<engine>/status`.

Boot sequence:
1. Load env
2. Seed admin (idempotent)
3. Mount 99 legacy routers (in order — priority modules first)
4. Bootstrap Mongo indexes for each engine (parallel-safe)
5. Emit boot log with router + engine status
6. Serve `/api/*`

## 18. Operations, Recovery, Runbook

### Common tasks

| Task | Command |
|------|---------|
| Restart backend | `sudo supervisorctl restart backend` |
| Tail backend log | `tail -f /var/log/supervisor/backend.err.log` |
| Force a learning cycle | `POST /api/learning/cycles/dispatch` (admin) |
| Force a meta-learning cycle | `POST /api/meta-learning/refresh?force=true` (admin) |
| Force a full portfolio rebuild | `POST /api/portfolio/rebuild/<master_bot_id>` (admin) |
| Engage broker kill switch | `POST /api/execution/broker/kill-switch` (admin) |
| Read orchestrator status | `GET /api/orchestrator/status` |
| Read all pending recommendations | `GET /api/meta-learning/pending` |
| Read execution journal range | `GET /api/execution/journal?start_seq=X&end_seq=Y` |

### Recovery

1. Kill switch first: `POST /api/execution/broker/kill-switch`.
2. Assess damage: `GET /api/execution/positions`, `GET /api/execution/risk/status`.
3. Cancel working orders: `DELETE /api/execution/orders/{request_id}` per open order.
4. Check journal for anomalies: `GET /api/execution/journal?event_type=...`.
5. Replay to reconstruct: `POST /api/execution/replay?start_seq=X&end_seq=Y`.
6. Rollback offending engine via env: e.g. `META_LEARNING_MODE=disabled` if meta-learning is suspect.
7. Restart backend: `sudo supervisorctl restart backend`.
8. Post-mortem: query `outcome_events` for the audit chain.

## 19. Validation Pyramid

Five tiers, each gated by exit code:

| Tier | Cadence | Runtime | Contents |
|------|---------|---------|----------|
| **Tier 1** | Every commit | ~2s | Memory backend + Phase H pytest + 10-order drill |
| **Tier 2** | Hourly | ~15s | Memory backend + 100-order clean + 500-order hostile stress |
| **Tier 3** | Daily | ~3min | Mongo backend + 500-order integration + full Phase A–I regression |
| **Tier 4** | Pre-release | ~10min | Mongo backend + 1000-order clean + 1000-order stress + full regression |
| **Tier 5** | Production | 24h / 72h | Continuous paper drill loop + aggregate JSON report |

`make ci-verdict` produces a one-line PASS/FAIL summary for pre-push hooks and GitHub commit status.

### Why five tiers instead of one?

Because CI cost and coverage are in tension. Tier 1 must run on every commit — anything longer than 5 seconds gets skipped. Tier 5 measures things (memory leak rate, dispatch drift, journal write throughput over 24h) that Tier 1 cannot see. Layering the tiers means every commit gets *some* coverage and every release gets *comprehensive* coverage.
