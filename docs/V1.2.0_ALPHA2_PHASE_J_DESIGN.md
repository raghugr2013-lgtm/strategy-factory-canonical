# Phase J — Factory Self-Evaluation

**Status:** DESIGN ONLY. Awaiting operator sign-off before implementation.

**Additive · backward compatible · OBSERVE MODE by default · instant rollback via feature flag.**

---

## 1. Objective

Phase I (Meta-Learning) closed the loop **at the engine level** — it evaluates specific decision surfaces (brain weights, thresholds, market weights) and recommends bounded parameter deltas. Phase I asks *"which knob should we turn?"*

Phase J operates **one level higher**. It evaluates the factory itself as an autonomous organisation, aggregating signals across every engine, every provider, every strategy family, every regime, every broker, every research investment. Phase J asks *"is the whole thing getting better, and if not, where is the bottleneck?"*

After Phase J, an operator (or a future automation layer) can answer:

* **Is the factory improving over time?** Rolling 7/30/90-day PnL, drawdown, prediction accuracy, execution quality, opportunity capture.
* **Which AI providers are performing best?** Cost-per-passing-strategy, quality-per-dollar, latency percentiles, availability, downstream realised PnL attributable to each provider.
* **Which research produced the highest value?** Which learning cycles, which prompt versions, which knowledge-base retrievals correlate with strategies that ultimately made money.
* **Which strategies contributed the most?** Marginal-portfolio-contribution ranking, not just solo PnL.
* **Which market regimes were handled well?** Regime-conditional Sharpe, hit-rate, and realised-vs-predicted delta by regime.
* **Which modules are becoming bottlenecks?** Orchestrator dispatch queue depth, learning-cycle throughput, backtest wait times, capacity-band residency, Mongo write latency.
* **Where should compute resources be shifted?** Per-task marginal value vs marginal cost; which tasks are starving; which are overspending.
* **Which research areas deserve more investment?** Style × regime × pair cells with high uncertainty (few samples) and high potential upside.
* **Which execution paths produced the highest quality?** Broker × pair × session × TIF combinations with lowest slippage-adjusted realised PnL delta.
* **Which portfolios are improving or deteriorating?** Per-Master-Bot health-trend regression, correlation drift, style-concentration drift.

Crucially, **Phase J recommends; it does NOT act.** In OBSERVE mode nothing downstream is mutated. No orchestrator priority changes, no budget reallocation, no strategy pruning, no portfolio rebuild forced. Phase J writes *insights* and *recommendations*; the operator (or future Phase K/L) decides when to apply them.

---

## 2. Design principles (non-negotiable — mirrors Phase I §2)

1. **OBSERVE MODE by default.** Phase J ships in a mode that cannot modify any active surface. Insights and recommendations are logged and API-readable; the wiring to apply exists but is dormant.
2. **Additive to the codebase.** New package `engines/factory_eval/`, new API prefix `/api/factory-eval/*`, new orchestrator task, new collections. Zero existing engine signatures change.
3. **Deterministic + explainable.** Every insight is produced by a documented heuristic with the exact input samples, sample sizes, and rationale attached.
4. **Bounded blast radius.** Even in future Recommend/Autonomous modes: per-surface caps, whitelisted targets, hard rate limits on recommendation generation.
5. **Full audit chain.** Every recommendation carries the outcome_events it aggregated (via immutable IDs) so any operator can walk back to the raw evidence.
6. **Idempotent + replayable.** Same input window + config → byte-identical recommendation IDs. Enables regression tests and post-mortem replay.
7. **Independent engine.** Phase J NEVER writes into any other engine's owned state. It **reads** from outcome_events + engine ledgers; it **writes** only to its own `factory_eval_*` collections.
8. **Factory-level scope.** Phase J does not duplicate Phase I. It aggregates *across* engines; Phase I focuses *within* an engine surface. The two are peers; both feed the operator dashboard.

---

## 3. Non-goals

* **NOT** a re-implementation of any existing engine.
* **NOT** an ML model trainer. Uses heuristics + rolling aggregations. Phase J.2 may add regression models later.
* **NOT** a UI. Backend autonomy remains the sole focus.
* **NOT** a scheduler replacement. Registers as one orchestrator task.
* **NOT** a duplicate of Phase I. Phase I evaluates engine-level surfaces; Phase J evaluates factory-level trends and reports.
* **NOT** a duplicate of Phase E. Phase E measures stability (memory, throughput, uptime); Phase J measures effectiveness (PnL, quality, cost efficiency).

## 4. Operating modes (mandatory per operator directive)

| Mode        | Value                | Behaviour                                                                                                                                          |
|-------------|----------------------|----------------------------------------------------------------------------------------------------------------------------------------------------|
| Disabled    | `disabled`           | Engine dormant. Orchestrator skips the task. Zero reads/writes.                                                                                    |
| **Observe** | `observe` (default)  | Engine runs on schedule. Reads outcome_events + ledgers. Computes reports + recommendations. Persists everything. **Applies nothing.**             |
| Recommend   | `recommend`          | Same as Observe, plus recommendations are surfaced in `/api/factory-eval/pending` and can be **explicitly applied by operator** via admin endpoint |
| Autonomous  | `autonomous`         | Whitelisted, low-blast-radius recommendations may be auto-applied (bounded delta, per-class cap). Belt-and-suspenders `AUTONOMOUS_CONFIRM=YES` env |

Mode transitions are **operator gated**: `disabled → observe → recommend → autonomous`. Every mode change emits a `factory_eval_mode_change` outcome event.

**Default configuration for this release: `FACTORY_EVAL_MODE=observe`.**

evidence_ref:         Dict[str, Any]   # cycle_id, n_outcome_events_scanned, etc

Every evaluator emits one or more insights. Insights are **observations**, not actions.

## 7. KPI catalogue (§ top-level factory metrics)

Each cycle computes the following into `FactoryReport.kpis`:

| KPI | Definition | Source |
|-----|------------|--------|
| `pnl_7d` / `pnl_30d` / `pnl_90d` | Rolling realised PnL | `execution_realised` outcome_events |
| `dd_current` / `dd_max_30d` | Current + max drawdown | Position roll-forward |
| `prediction_accuracy_30d` | Mean of `1 − |predicted − realised|` over closed trades | `execution_attribution` |
| `execution_quality_p50/p95` | Quality score distribution | `execution_quality` collection |
| `opportunity_capture_ratio` | Realised PnL / (theoretical-max PnL from all pending strategies) | Cross-engine join |
| `ai_spend_daily / _monthly` | AI provider spend | orchestrator budget tracker |
| `ai_spend_per_passing_strategy` | Total spend / strategies that passed backtest | Cross-join |
| `orchestrator_dispatches_1h / 24h` | Task dispatch counters | orchestrator internal |
| `learning_cycles_per_hour` | Continuous scheduler throughput | scheduler stats |
| `capacity_band_time_share` | % time spent in healthy / warn / critical bands | adaptive_concurrency |
| `mongo_pool_used_p95` | Mongo connection pool pressure | motor client stats |
| `queue_pressure_p95` | Task queue depth pressure | queue_pressure module |
| `router_count` | Boot-log router count invariant | log scan |
| `orchestrator_task_count` | Registered tasks | registry |
| `structural_changes_24h` | Phase G structural changes detected | `structural_changes` |
| `broker_health_score_p50` | Rolling broker-health score | `broker_health` |
| `attribution_coverage_pct` | % closed positions with attribution rows | Cross-join |
| `meta_learning_pending_count` | Phase I pending recommendations | Phase I ledger |

All KPIs are **derived facts**, computed once per cycle, persisted in the FactoryReport. Downstream consumers query the report, not the raw sources — this is critical for consistency across dashboard views.

## 8. Evaluator heuristics

Every evaluator is a pure function of read-only inputs. Same pattern as Phase I: deterministic, unit-testable in isolation, replay-safe.

### 8.1 `factory_improvement` — is the whole thing getting better?

For each KPI in {`pnl_daily`, `prediction_accuracy`, `execution_quality`, `opportunity_capture`}:
1. Compute rolling 7d, 30d, 90d means.
2. Fit a simple linear regression `y = a + b × t` over the window.
3. Emit insight `FactoryInsight(surface="factory_improvement", target=kpi, metrics={slope, r2, mean_current, mean_prior})`.
4. If `|slope| ≥ threshold` AND `r² ≥ 0.30`, mark `severity=low|med|high`. Otherwise `severity=info`.

### 8.2 `provider_efficiency` — which AI providers deliver value?

For each provider in the ai_workforce_router registry:
1. Sum `cost_usd` from `outcome_events` where `provider=<X>` in the window.
2. Count strategies with `learning_run_id` traceable to this provider that ultimately `backtest_passed`.
3. Cost-per-passing-strategy = spend / count (+ ε).
4. Sum realised PnL attributable to strategies with this provider on their lineage.
5. Emit per-provider insight with `metrics={cost, count_passing, cost_per_pass, realised_pnl, roi}`.

### 8.3 `research_roi` — which research produced the highest value?

For each `learning_run_id` in the window:
1. Cost = sum `cost_usd` across all outcome_events with matching run_id.
2. Value = sum realised PnL from strategies with this run_id on their lineage that ultimately traded.
3. ROI = (value − cost) / (cost + ε).
4. Group by prompt_version / model / knowledge_context_hash to detect systematic winners/losers.
5. Emit insights per group with `metrics={cost, value, roi, n_strategies}`.

### 8.4 `strategy_ranking` — marginal contribution

For each active strategy `s` in the last window:
1. Read realised PnL for trades initiated by `s`.
2. Read portfolio-level PnL variance contribution using standard risk-parity formula.
3. Marginal contribution = solo PnL × (1 − avg_pairwise_correlation).
4. Rank all strategies by marginal contribution.
5. Emit insight per top-N (default 30) and bottom-N (default 30).

### 8.5 `regime_effectiveness` — per-regime scoring

For each regime in `{trending, ranging, high_volatility, low_volatility}`:
1. Filter `execution_realised` events by decision-time regime.
2. Compute realised Sharpe (mean/std of realised PnL), hit-rate, mean `delta_predicted_realised`.
3. Emit per-regime insight with `metrics={sharpe, hit_rate, mean_delta, n_samples}`.

### 8.6 `bottleneck_detector` — where are we saturated?

For each observable saturation signal:
- Orchestrator `dispatch_queue_depth > threshold` for > 10% of ticks.
- Adaptive concurrency in `critical` band > 5% of window.
- Mongo write p95 > threshold for > 5 minutes.
- Learning cycle wait time (queued → running) > threshold p95.

Emit insight per detected bottleneck with `severity` proportional to time-in-saturated-state.

### 8.7 `compute_allocation` — should we shift resources?

For each orchestrator task class:
1. Compute marginal_value_per_second = business_value × realised_pnl_attributed / cpu_seconds_consumed.
2. Rank tasks by marginal value.
3. If a passive-by-default task has high marginal value estimate, emit recommendation to lower its `priority_base` gap OR promote it.
4. If an active task has low marginal value, emit recommendation to lower priority OR mark passive.

**Note**: this is a *hint*, not an autonomous action in Phase J. Even in Recommend mode, changing orchestrator task priorities requires operator approval.

### 8.8 `execution_quality_ranking` — which paths are best?

For each `(broker, pair, session, time_in_force)` tuple with ≥ N samples:
1. Compute mean realised_execution_score, mean slippage, mean latency.
2. Rank paths within each `(broker, pair)` group.
3. Emit insights identifying top-3 and bottom-3 paths per pair.

### 8.9 `portfolio_health_trends` — per-Master-Bot regression

For each Master Bot with ≥ 30 days of recorded state:
1. Compute rolling 7d Sharpe, drawdown, style-concentration entropy.
2. Fit linear regression on trailing 30d.
3. Emit insight per bot with `metrics={sharpe_slope, dd_slope, style_entropy_slope}`.
4. `severity=high` if all three slopes are unfavourable.

### 8.10 `coverage_gap_detector` — where should we invest research?

For every `(style, regime, pair)` cell:
1. Count strategies that have ever traded in that cell.
2. Count decisions taken in that cell.
3. If count < threshold AND cell has potential (e.g. recent structural change in that regime), emit `research_investment` opportunity.

## 9. Proposers (evaluation → recommendation)

Each proposer consumes FactoryInsights and emits FactoryRecommendations. Bounded by:
- `FACTORY_EVAL_MAX_DELTA_PER_TICK` on any numeric target.
- Per-surface `class_caps` on cumulative delta per rolling 24h.
- Whitelist for autonomous mode.

Concrete proposers:

* **compute_reallocation** — bounded deltas to `ORCH_TASK_<NAME>_PRIORITY_BASE`; only when marginal-value gap ≥ threshold.
* **budget_reallocation** — bounded shifts of AI provider daily budget (only *proportional* shifts, never absolute cuts below a floor).
* **research_investment** — a *hint*: which (style, regime, pair) cells have highest expected uplift for the next learning cycle. Not a mutation — just a ranked list that Phase K could consume.
* **strategy_pruning** — identifies strategies in the lowest marginal-contribution decile with sustained losses. **In OBSERVE these are recommendations only**; even in Autonomous mode, actual retirement decisions still go through Phase D's retirement engine (Phase J only recommends the candidate list).
* **portfolio_rebalance_hint** — flags Master Bots trending unfavourably. Does not force a rebuild — just surfaces the finding.
* **execution_path_pref** — recommends `EXEC_PREFERRED_PATH_<PAIR>` env for future orders. Downstream execution engine reads this only if `EXEC_USE_FACTORY_EVAL_OVERRIDES=true`.

## 10. Interaction with Phase I

**Peers, not consumers.** Both engines read `outcome_events`. Both write to their own ledgers. Both surface recommendations to the operator.

Phase I focuses on the **numeric levers inside engines** (weights, thresholds, gates). Phase J focuses on the **factory-wide investment decisions** (compute, budget, research, strategy composition, execution paths, portfolio evolution).

They do **not** co-sign in Phase J's initial release (this was Q7 of the Phase I design, deferred). Each engine ranks independently; the operator sees both lists side-by-side.

Future integration point (Phase K): a *meta-meta-learner* could combine Phase I + Phase J signals into a single ranked "actions to take today" list. That is explicitly out of scope now.

## 14. Explainability chain

Every factory-eval cycle produces a chain of outcome_events:

```
factory_eval_cycle_start
  ├─ factory_report                       (one per cycle — the aggregate snapshot)
  ├─ factory_eval_insight                 (one per evaluator finding)
  │    └─ evidence: outcome_events read, ledger rows aggregated
  ├─ factory_eval_recommendation          (one per proposal)
  │    └─ evidence: insight_ids, ranker_score, guardrails
  └─ factory_eval_cycle_end
       └─ summary: n_insights, n_recommendations, mode, duration_ms
```

In future modes:
```
factory_eval_recommendation
  └─ factory_eval_application             (only when applied)
       └─ factory_eval_revert             (only when reverted)
factory_eval_mode_change                  (on mode transition)
```

Every event carries joining IDs (`report_id`, `insight_id`, `recommendation_id`, `application_id`, `learning_run_id` where applicable). Full audit chain resolvable via outcome_events alone.

## 17. Rollback strategy

1. **Instant**: `FACTORY_EVAL_MODE=disabled` — orchestrator skips the task; API still serves historical reads.
2. **Task disable**: `ORCH_TASK_FACTORY_EVALUATION_PASSIVE=true`.
3. **Downstream disable**: `<ENGINE>_USE_FACTORY_EVAL_OVERRIDES=false` (default) — nothing consumes overrides.
4. **Delete overrides**: `POST /api/factory-eval/overrides/{target}/revert`.
5. **Nuclear**: drop the six `factory_eval_*` collections. Every other engine continues byte-identically.

## 21. OPEN OPERATOR QUESTIONS (please answer before implementation begins)

### **Q1 — Cadence and window strategy**
Two cadences proposed: **regular hourly cycle** (24h window) + **daily 03:00 UTC report** (90d window). The regular cycle keeps operators informed intra-day; the daily report captures long-term trends.

* **Option A** — Hourly + daily (default proposal)
* **Option B** — Only hourly with adaptive window (starts at 24h, extends automatically)
* **Option C** — Only daily 03:00 UTC (simplest, but no intra-day visibility)

Recommendation: **A**.

### **Q2 — Autonomous whitelist**
Which surfaces should ever be eligible for auto-application in a *future* Autonomous mode?

* [ ] `compute_reallocation` — orchestrator priority nudges
* [ ] `budget_reallocation` — AI provider budget shifts (proportional only)
* [ ] `research_investment` — hint-only (never auto-applied, always operator-reviewed)
* [ ] `strategy_pruning` — flag candidates only; retirement still runs through Phase D
* [ ] `portfolio_rebalance_hint` — never auto-applied (Phase D owns rebuilds)
* [ ] `execution_path_pref` — path preference for future orders
* [ ] **None** — Autonomous never enabled; recommendations always require operator approval

Recommendation: **`compute_reallocation` + `execution_path_pref` only**. Everything else stays operator-gated indefinitely.

### **Q3 — Interaction with Phase I**
Should Phase J and Phase I co-sign recommendations?

* **Option A** — Independent (each surfaces its own recommendations)
* **Option B** — Co-signed only (recommendation shown only if both engines agree)
* **Option C** — Independent for now; add co-signing in Phase K

Recommendation: **C**.

### **Q4 — Report retention**
`factory_eval_reports` proposed TTL: 180 days. Insights: 90 days. Recommendations: 180 days. Applications: 365 days.

* Is 180 days enough for reports? (Some auditors want 1 year+.)
* Should we archive reports older than TTL to `factory_eval_reports_archive` (permanent) before deletion?

Recommendation: **archive to `_archive` before TTL delete**, so full history is queryable forever.

### **Q5 — KPI floor: which KPIs are load-bearing?**
The KPI catalogue proposes ~20 KPIs. Some are more critical than others. Which are the "P0" KPIs that every dashboard view should always show?

Recommendation (P0 minimum set):
1. `pnl_30d`
2. `dd_current`
3. `prediction_accuracy_30d`
4. `execution_quality_p50`
5. `ai_spend_daily`
6. `opportunity_capture_ratio`

Rest are P1 (drill-down views only).

### **Q6 — Bottleneck sensitivity**
The bottleneck_detector fires when any saturation signal exceeds thresholds. This can be noisy. Two options:

* **Option A** — Emit an insight per bottleneck (potentially many per cycle)
* **Option B** — Emit a single "bottleneck summary" insight per cycle listing top-3

Recommendation: **B** — reduces alarm fatigue while preserving all detail in the summary.

### **Q7 — Provider leaderboard granularity**
* **Option A** — Per-provider (openai, anthropic, gemini, etc.)
* **Option B** — Per (provider, model) tuple (e.g. openai/gpt-5.2, anthropic/claude-sonnet-4.6)
* **Option C** — Both — high-level provider view + drill-down to per-model

Recommendation: **C**.

### **Q8 — Coverage gap threshold**
`coverage_gap_detector` fires when `(style, regime, pair)` has < N strategies AND recent structural change in that regime. What is N?

Recommendation: **N = 3** — a cell with fewer than 3 strategies is under-explored; more than that we consider covered.

### **Q9 — Compute reallocation risk**
The `compute_reallocation` proposer would recommend nudging `ORCH_TASK_<NAME>_PRIORITY_BASE`. A recommendation might, for example, propose lowering `learning_cycle` priority by 2 points. This is a numeric change to a running orchestrator.

* **Option A** — Recommend-only; always requires operator approval (default proposal)
* **Option B** — Autonomous with per-day cap of ±3 priority points
* **Option C** — Autonomous only after 30-day OBSERVE validation shows recommendations converge

Recommendation: **A** — orchestrator priority is the factory's central nervous system; changes should be conscious.

### **Q10 — Strategy pruning gate**
`strategy_pruning` flags candidates. Even if autonomous is enabled, actual retirement runs through Phase D. Should Phase J's flagging cause Phase D to escalate the retirement decision?

* **Option A** — Phase J flags in factory_eval_insights only; Phase D unchanged
* **Option B** — Phase J's flag adds a `factory_eval_flagged=true` field to the Phase D retirement input, letting Phase D's own logic decide
* **Option C** — Defer decision until Phase D + Phase J are both operationally validated

Recommendation: **A** for this release. **C** as the long-term plan.
