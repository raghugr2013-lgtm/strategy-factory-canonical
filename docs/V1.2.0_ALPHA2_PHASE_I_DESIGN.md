# Phase I ŌĆö Meta-Learning

**Status:** DESIGN ONLY. Awaiting operator sign-off before implementation.

**Additive ┬Ę backward compatible ┬Ę OBSERVE MODE by default ┬Ę instant rollback via feature flag.**

---

## 1. Objective

Phases AŌĆōH taught the factory to **generate**, **classify**, **allocate**, **decide**, **read the market** and **execute**. Phase I closes the outer feedback loop: **learn from every decision the factory has already made and propose how the factory itself should evolve.**

Today the brain scorer, portfolio allocator, activation thresholds, market-intelligence weights and execution-quality weightings are all **hand-tuned** via env vars (e.g. `BRAIN_W_REGIME_FIT=0.20`, `BRAIN_TRADE_NOW_THRESHOLD=0.75`). Operators pick sensible starting points; nothing measures whether those numbers were correct.

Phase I introduces a dedicated **Meta-Learning Engine** that continuously:

1. **Collects** every `brain_decision`, `execution_realised`, `execution_quality_refresh`, `market_intelligence_refresh`, `structural_change_detected`, `portfolio_rebuild`, `activation`, `promotion`, and `retirement` outcome_event.
2. **Evaluates** how well each *decision surface* (scoring weights, thresholds, market/execution/portfolio inputs) predicted realised outcomes.
3. **Calculates** proposed weight adjustments using deterministic, explainable heuristics (no black-box ML in Phase I).
4. **Ranks** recommendations by expected marginal improvement, statistical significance and blast radius.
5. **Persists** every recommendation to an immutable `meta_learning_recommendations` collection with a full audit chain.
6. **Emits** outcome events (`meta_learning_evaluation`, `meta_learning_recommendation`) so downstream engines and future audits can trace exactly why each recommendation was produced.

After Phase I the factory can answer:

* **Which scoring weight has been under- or over-shooting realised PnL over the last N decisions?**
* **Is `BRAIN_TRADE_NOW_THRESHOLD=0.75` still the right threshold, or has drift moved it?**
* **Which style-x-regime combinations are systematically miscalibrated?**
* **Do we need more or less weight on Phase G's market-confidence signal?**
* **What is the smallest single change that would deliver the largest expected uplift right now?**

Crucially, **Phase I proposes**; it does **NOT act**. In OBSERVE mode nothing downstream is mutated ŌĆö no brain weight, no threshold, no portfolio weight, no ranking, no live execution behaviour changes. Operators (or future Phase J) decide when a recommendation is actually applied.

---

## 2. Design principles (non-negotiable)

1. **OBSERVE MODE by default.** Phase I ships in a mode that *cannot* modify any active surface. Recommendations are logged and API-readable; the wiring to apply them exists but is dormant.
2. **Additive to the codebase.** New package `engines/meta_learning/`, new API prefix `/api/meta-learning/*`, new orchestrator tasks, new collections. Zero existing module signatures change. `engines/brain/`, `engines/portfolio/`, `engines/execution/`, `engines/market_intel_engine/` stay byte-identical unless an operator later opts-in to autonomous mode.
3. **Deterministic + explainable.** No opaque ML models in Phase I. Every recommendation is produced by a documented heuristic with the exact input samples, sample sizes, and rationale attached (`evidence` block on every recommendation row).
4. **Bounded blast radius.** Even in future Recommend/Autonomous modes: max delta per tick per weight, per-class caps, and hard whitelists of which surfaces may be modified.
5. **Full audit chain.** Every recommendation carries: (a) the outcome_events it read (immutable IDs), (b) the metrics it computed, (c) the proposed delta, (d) the confidence-in-recommendation score, (e) the surface it targets, (f) the mode under which it was produced.
6. **Idempotent + replayable.** Given the same input window and config, meta-learning produces byte-identical recommendations. Enables regression tests and deterministic replay.
7. **Independent engine.** Meta-learning NEVER writes into brain / portfolio / execution engine state. Any activation happens through explicit application steps recorded in a separate `meta_learning_applications` collection (empty until Autonomous mode is enabled).

---

## 3. Non-goals

* **NOT** a re-implementation of the brain, portfolio or execution engines.
* **NOT** an ML model trainer. Phase I uses linear/heuristic evaluators; Phase I.2 (later) may add regression-based fits.
* **NOT** a UI. Backend autonomy remains the sole focus.
* **NOT** a strategy-optimiser or hyperparameter search. It optimises *the factory's decision-making meta-parameters*, not the strategies themselves.
* **NOT** a scheduler replacement. Phase I plugs into the existing Unified Orchestrator (Phase B.2) as a subordinate task.
* **NOT** a new persistence framework. Uses the same Mongo collections/patterns as prior phases.

---

## 4. Operating modes (mandatory per operator directive)

Phase I supports four operating modes, selected via `META_LEARNING_MODE`:

| Mode        | Value          | Behaviour                                                                                                                                                            |
|-------------|----------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Disabled    | `disabled`     | Engine dormant. Orchestrator skips the task. Zero reads, zero writes. Zero performance impact.                                                                       |
| **Observe** | `observe` (default) | Engine runs on schedule. Reads outcome_events. Computes evaluations. Persists recommendations. Emits outcome events. **Applies nothing.** API is read-only.        |
| Recommend   | `recommend`    | Same as Observe, plus recommendations are surfaced in `/api/meta-learning/pending` and can be **explicitly applied by operator** via authenticated admin endpoint.  |
| Autonomous  | `autonomous`   | Recommendations that pass the auto-apply whitelist + confidence gate are applied automatically (bounded delta, per-class cap). Every application is journaled + reversible. |

Mode transitions are **operator gated**: `disabled ŌåÆ observe ŌåÆ recommend ŌåÆ autonomous`. Downgrading is always allowed at runtime (env change + service reload). Every mode change is emitted as `meta_learning_mode_change` outcome event.

**Default configuration for this release: `META_LEARNING_MODE=observe`** (per operator instruction).

## 7. Evaluator heuristics (┬¦ deterministic)

Every evaluator is a **pure function** of read-only Mongo inputs ŌĆö no state, no side effects except the returned `MetaEvaluation`. This makes them unit-testable in isolation and byte-identical across replays.

### 7.1 `weight_sensitivity` ŌĆö brain scoring weights

For each scoring weight `w_i` (regime_fit, confidence, recent_pf, long_pf, dd_penalty, prediction_acc, corr_penalty, session_fit, liquidity_fit, market_confidence, style_confidence, opportunity):

1. Read all `brain_decision` outcome_events in the window with a matched `execution_realised` (via `brain_decision_id`).
2. For each decision `d`, retrieve `d.metrics.components[i]` (the weighted contribution of `w_i` at decision time) and the paired realised outcome (`realised_pnl` normalised to `[-1, 1]`).
3. Compute Pearson correlation `Žü_i` between component contribution and realised outcome across Ōēź `META_LEARNING_MIN_SAMPLES` (default 50) decisions.
4. Compute `expected_uplift_i = clamp(Žü_i ├Ś step_size, -max_delta, +max_delta)` where `step_size = META_LEARNING_WEIGHT_STEP` (default 0.01).
5. Emit `MetaEvaluation(surface="brain_weight", target="BRAIN_W_<i>", metrics={"correlation": Žü_i, "expected_uplift": expected_uplift_i}, significance=|Žü_i|)`.

If `|Žü_i| Ōēź META_LEARNING_SIG_THRESHOLD` (default 0.20) **and** `n_samples Ōēź min_samples`, a `MetaRecommendation` is proposed with `proposed_delta = expected_uplift_i` (sign-preserving), classified as `risk_band=green` (bounded, Ōēżmax_delta), `severity=info|low|med` based on `|Žü_i|` band.

### 7.2 `threshold_calibration` ŌĆö action thresholds

For `BRAIN_TRADE_NOW_THRESHOLD`, `BRAIN_PAUSE_THRESHOLD`, `BRAIN_RETIRE_THRESHOLD`:

1. Bucket all decisions by `score_now` into 20 equal-width bins.
2. Compute realised win-rate + realised mean-PnL per bin.
3. Find the "optimal" threshold `t*` maximising `E[PnL | score_now Ōēź t*] ├Ś count(score_now Ōēź t*)` subject to minimum trade count.
4. Recommend `proposed_delta = clamp(t* ŌłÆ current, -0.05, +0.05)`.

### 7.3 `confidence_calibration` ŌĆö brain confidence reliability

1. Bucket decisions by `confidence` (10 bins).
2. Compute mean realised outcome per bin.
3. Compute reliability gap: `|expected_outcome(bin) ŌłÆ realised_outcome(bin)|`.
4. If mean gap Ōēź `META_LEARNING_CALIB_GAP_MIN` (default 0.10), recommend either (a) an overall shrinkage factor `CONFIDENCE_SHRINK` (never applies to raw strategy backtest confidence, only to brain-time overlays), or (b) a per-style shrinkage vector (Phase I.2 promotion).

### 7.4 `style_regime_matrix` ŌĆö 6├Ś4 miscalibration matrix

For each cell `(style, regime)`:

1. Count decisions in cell, realised PnL, expected score at decision time.
2. Compute normalised realised score.
3. Cell miscalibration = `mean_realised ŌłÆ mean_expected`.
4. Recommend a per-cell fit-table adjustment (currently a proposal only; would land in `_STYLE_REGIME_FIT` ŌĆö an override table, NEVER the constants in `scorer.py`).

### 7.5 `market_signal_utility` ŌĆö Phase G weights

Same shape as 7.1 but restricted to the three additive Phase G weights (`BRAIN_W_MARKET_CONFIDENCE`, `BRAIN_W_STYLE_CONFIDENCE`, `BRAIN_W_OPPORTUNITY`), each of which is `0.0` by default. Recommendation includes a **first-activation gate**: even if `Žü Ōēź threshold`, the initial recommendation is capped at `0.05` (never a large jump from `0.0`).

### 7.6 `execution_quality_gate` ŌĆö Phase H realised uplift

Uses `execution_attribution.delta_predicted_realised` distribution:

1. If p95 negative delta Ōēż ŌłÆ0.15 for Ōēź N=30 attributions on a broker/pair combo, recommend `execution_gate` action = tighten `EXEC_QUALITY_MIN_SCORE`.
2. If distribution centered around 0 (well-calibrated), emit `info` evaluation with no recommendation.

## 8. Ranker

The ranker takes all `MetaRecommendation` candidates in a cycle and produces a ranked shortlist.

**Score = `expected_uplift ├Ś confidence ├Ś (1 ŌłÆ risk_penalty(risk_band))`**

Where:
* `risk_penalty("green") = 0.0`
* `risk_penalty("amber") = 0.5`
* `risk_penalty("red") = 1.0` (effectively suppressed)

Ties broken by lowest cumulative-application delta on that target in the last 24h (encourages breadth over repeated hammering of one weight). Recommendations below `META_LEARNING_RANK_FLOOR` (default 0.01) are marked `status="expired"` and remain in the ledger for audit but are hidden from `/api/meta-learning/pending`.

## 9. Applier (dormant in OBSERVE ŌĆö designed for future modes)

`applier.py` implements a **single narrow surface**: it writes to a dedicated Mongo collection `meta_learning_overrides` **only** in Recommend (via operator API) or Autonomous mode. It never edits env vars, never mutates other engines' state.

Downstream consumers *opt in* to reading these overrides:

* `brain/config.py::scoring_weights()` optionally reads `meta_learning_overrides` if `BRAIN_USE_META_OVERRIDES=true` (default `false`).
* Ditto for `trade_now_threshold()`, `pause_threshold()`, etc.

**In OBSERVE mode**, `BRAIN_USE_META_OVERRIDES` is `false`, `META_LEARNING_MODE=observe`, so this file path is unreachable ŌĆö the applier is instantiated but never invoked. Extensive unit tests assert that in OBSERVE mode a full cycle produces zero writes to `meta_learning_overrides` and zero writes to `meta_learning_applications`.

**Rollback** is trivial: delete the entries in `meta_learning_overrides` (or flip the `BRAIN_USE_META_OVERRIDES` flag to `false`). Every application row records `previous_value`, so revert is a simple lookup.

## 16. Rollback strategy

1. **Instant**: `META_LEARNING_MODE=disabled` ŌĆö orchestrator skips the task; API still serves historical reads.
2. **Full disable**: unregister the orchestrator task via env `ORCH_TASK_META_LEARNING_EVALUATION_PASSIVE=true`.
3. **Delete overrides** (future modes only): `POST /api/meta-learning/overrides/{target}/revert`.
4. **Nuclear**: drop the four `meta_learning_*` collections. All other engines continue running byte-identically.

No other engine ever changes behaviour based on Phase I being on or off, unless an operator explicitly sets `*_USE_META_OVERRIDES=true`.

## 25. OPEN OPERATOR QUESTIONS (please answer before implementation begins)

### **Q1 ŌĆö Evaluation window(s)**
Default `META_LEARNING_WINDOW_HOURS=24`. This gives fast feedback but small sample sizes early.
* **Option A** ŌĆö Single 24h window (default proposal).
* **Option B** ŌĆö Cascade of windows (1h / 24h / 7d). Recommendation only fires when all three agree in direction. Slower but higher-confidence.
* **Option C** ŌĆö Adaptive window: start at 24h; extend automatically when `n_samples < min_samples`.

Recommendation: **B** for extra caution given OBSERVE-mode learning phase, then loosen to A once VPS Tier 5 validation completes.

### **Q2 ŌĆö Which surfaces are eligible for future *autonomous* mode?**
Even if Autonomous mode is never enabled in this release, the whitelist determines what *could* be automated later. Please tick the surfaces you would consider:
* [ ] `BRAIN_W_*` (brain scoring weights) ŌĆö small, bounded deltas per tick
* [ ] `BRAIN_TRADE_NOW / PAUSE / RETIRE` thresholds
* [ ] `BRAIN_W_MARKET_* / _STYLE_ / _OPPORTUNITY` (Phase G weights)
* [ ] Portfolio style-cap / correlation-cap
* [ ] `EXEC_QUALITY_MIN_SCORE` / execution gate
* [ ] **None** ŌĆö Autonomous mode will remain permanently disabled; recommendations always require operator approval

Recommendation: **BRAIN_W_\*** and **Phase G weights** only. Thresholds and portfolio caps remain permanently manual until operator explicitly promotes them.

### **Q3 ŌĆö Recommendation TTL and duplicate suppression**
* **Option A** ŌĆö Recommendations expire in 7 days; identical (surface, target) proposals suppressed for 24h to avoid noise.
* **Option B** ŌĆö Recommendations expire in 24h; no suppression (always freshest view).
* **Option C** ŌĆö No expiry; operator manually rejects/approves.

Recommendation: **A**.

### **Q4 ŌĆö Minimum samples per evaluator**
Default `META_LEARNING_MIN_SAMPLES=50`.
* Is 50 realistic against your expected daily decision volume?
* Should we start at 30 during warmup (first two weeks) then step to 50?

Recommendation: **30 during warmup ŌåÆ 50 steady-state**, driven by `META_LEARNING_WARMUP_UNTIL` (ISO date env).

### **Q5 ŌĆö Statistical significance floor**
`META_LEARNING_SIG_THRESHOLD=0.20` (Pearson `|Žü|`). Below this, no recommendation is emitted even if a delta is technically calculable.
* Is 0.20 acceptable?
* Should we require **both** `|Žü| Ōēź 0.20` AND `p-value < 0.05` (Spearman rank-corr as a robustness check)?

Recommendation: **Both**. Slower to fire; much cleaner recommendations.

### **Q6 ŌĆö Mode-change authorisation**
