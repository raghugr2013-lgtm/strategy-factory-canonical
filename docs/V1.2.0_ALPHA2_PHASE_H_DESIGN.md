# Phase H — Execution Intelligence

**Status:** DESIGN ONLY. Awaiting operator sign-off before implementation (same protocol as Phases F & G).

**Additive · backward compatible · instant-rollback via feature flag.**

---

## 1. Objective

Phase A–G taught the factory to learn **from strategy performance** (A–F) and **from the market itself** (G). Phase H closes the third feedback loop: **learn from live execution**.

Today `engines/brain/execution_quality.py` ESTIMATES execution quality from optional metadata (spread / latency / slippage / rejects / broker health / fill quality) and defaults every missing input to a neutral `0.7`. That was Phase F's Q4 provisional design.

Phase H replaces the estimator with a live-measurement pipeline fed by the cTrader Open API. The Adaptive Trading Brain then consumes **realised** execution quality per broker, per pair, per session, per strategy — and the closed-learning layer can attribute realised PnL back to specific brain decisions.

After Phase H the factory can answer:

* **How wide is my spread on EURUSD during the NY overlap on IC Markets today?**
* **Which strategies suffer the most slippage during high-volatility sessions?**
* **Is the broker connection healthy right now, or should risk be reduced?**
* **Did the brain's `TRADE_NOW` decisions actually get filled at the expected price?**

The last question closes the loop: Phase F's `brain_decision` outcome events now have a matching `execution_realised` twin — every prediction gets scored.

---

## 2. Non-goals

* **NOT** a full production trading gateway. Phase H is a read-mostly execution feed + a controlled order-submission surface used only by operator-driven flows. Auto-trading remains gated behind existing risk switches.
* **NOT** a broker-agnostic execution layer. Phase H targets cTrader Open API (the operator's live venue). A `BrokerAdapter` protocol is added so Interactive Brokers / MT5 / others can slot in later without refactoring.
* **NOT** meta-learning of brain weights (Phase I) — but every `execution_realised` row is tagged so Phase I can train on it directly.
* **NOT** a UI. Backend autonomy remains the sole focus.
* **NOT** a re-implementation of the strategy engine. Order requests still originate from existing decision engines; Phase H only measures + records + attributes.

## 5. Persistence

New Mongo collections (all additive, indexed):

| Collection | Purpose | Key indexes |
|---|---|---|
| `order_requests` | Every submitted order + client idempotency key | `(request_id)` unique, `(strategy_hash, requested_at DESC)` |
| `fill_events` | Every fill (partial + final) | `(request_id)`, `(pair, timestamp DESC)`, `(fill_id)` unique |
| `positions` | Live + closed positions | `(position_id)` unique, `(pair, opened_at DESC)`, `(closed_at)` sparse |
| `broker_health` | Rolling health rows | `(broker, ts DESC)` |
| `execution_quality` | Rolling per-pair×session snapshots | `(pair, session, window, ts DESC)` |
| `execution_attribution` | brain-decision ↔ fill joins | `(brain_decision_id)`, `(strategy_hash, fill_ts DESC)` |

All writes route through `ledger.py`. No other module writes to these collections (single source of truth).

TTLs:
* `broker_health` → 30 days (`expires_at` field)
* `execution_quality` → 90 days
* `order_requests` / `fill_events` / `positions` / `execution_attribution` → **no TTL** (audit trail is permanent)

## 6. Broker adapter contract

`BrokerAdapter` (Protocol — pluggable, zero cost duck typing):

```python
class BrokerAdapter(Protocol):
    NAME: str
    async def connect(self) -> None: ...
    async def disconnect(self) -> None: ...
    async def submit(self, req: OrderRequest) -> str: ...        # returns broker order id
    async def cancel(self, request_id: str) -> bool: ...
    async def positions(self) -> List[Position]: ...
    async def stream_fills(self) -> AsyncIterator[FillEvent]: ...
    async def health(self) -> BrokerHealth: ...
```

Implementations:

1. **`PaperBrokerAdapter`** (default, `BROKER=paper`)
   * Deterministic paper fills — configurable slippage / rejection / partial rates.
   * Used for CI, local dev, and the entire Phase H test suite.
2. **`CtraderBrokerAdapter`** (`BROKER=ctrader`)
   * cTrader Open API v2 (Protobuf over websocket + REST OAuth).
   * OAuth token stored in Mongo `broker_credentials` collection encrypted at rest.
   * Websocket connection managed by `resilience.py` — see §7.

The operator picks a broker with `BROKER=ctrader|paper` env var. Default `paper` so Phase H is safe to boot without cTrader credentials.

## 7. Connection resilience (§operator brief)

`resilience.py` handles the failure surface of any live venue:

* **Exponential backoff** (100ms → 30s cap) on connection failure.
* **Circuit breaker** — 5 consecutive failures in 60s → OPEN for 5min → HALF-OPEN test → CLOSED.
* **Heartbeat** — every 15s while connected; missed 3 heartbeats → force reconnect.
* **Session refresh** — cTrader OAuth token TTL ≈ 60min; a subordinate orchestrator task `broker_session_refresh` renews 10min before expiry.
* **Duplicate suppression** — every `submit` carries a `request_id`; retries after a network glitch never place a duplicate order (broker-side dedup by client-request-id).
* **Kill switch** — `BROKER_KILL_SWITCH=true` disconnects immediately, cancels all working orders, blocks any new submit until manually cleared. Also fires on `risk_monitor` breach (§11).

Health scoring (rolling 60 min):

```
score = 0.35 * uptime_pct
      + 0.25 * (1 - reject_rate)
      + 0.15 * (1 - requote_rate)
      + 0.15 * latency_norm            # 1.0 @ ≤50ms → 0.1 @ 1000ms
      + 0.10 * (1 - disconnect_bias)   # disconnects/hour normalised
```

Band mapping: `≥0.8 → healthy`, `0.5..0.8 → degraded`, `<0.5 → unhealthy`.

## 8. Order lifecycle

```
PENDING → SENT → (ACCEPTED | REJECTED)
                    ↓
                 WORKING → (PARTIAL_FILL* → FILLED | CANCELLED | EXPIRED)
```

`order_lifecycle.py`:

* Every state transition writes an `outcome_events` row (`order_state_change` decision type).
* Terminal states are `FILLED`, `REJECTED`, `CANCELLED`, `EXPIRED`.
* Partial fills accumulate on the same `request_id` — each writes a `FillEvent` row and updates the working quantity.
* Timeouts: `MARKET` orders that don't ACK within `EXEC_MARKET_ACK_TIMEOUT_MS` (default 2000ms) are auto-cancelled and reported as `REJECTED{reason=ack_timeout}`.

---

## 9. Position lifecycle

```
OPEN → (PARTIAL_CLOSE*) → CLOSED
```

* Positions are opened on `FILLED` and closed on the reverse-side full fill.
* PnL is realised on each partial close; unrealised PnL is refreshed on every price tick and stored inline (not to Mongo per-tick — that lives in memory).
* Every close writes an `outcome_events` row (`position_closed` decision type) with realised PnL + `strategy_hash` + `brain_decision_id` — feeds closed learning directly.

## 10. Execution quality — from ESTIMATED to MEASURED

`quality.py` exposes:

```python
async def measure_execution_quality(
    pair: str, session: Optional[str] = None,
    window: str = "24h",
) -> ExecutionQualitySnapshot
```

Reads recent `fill_events` (matched to `order_requests`) and computes the six-component composite with the same weights Phase F's estimator uses (`spread 0.30 / latency 0.20 / slippage 0.20 / rejects 0.15 / broker 0.10 / fill 0.05`). `method="measured_live"` — the string is the signal the brain uses to know it can trust the number.

`estimate_execution_quality()` is preserved. `brain/execution_quality.py` gains a switch:

```python
if execution_config.live_measurement_enabled() and has_recent_fills(pair):
    eq = await measure_execution_quality(pair, session)
else:
    eq = estimate_execution_quality(**metadata)   # Phase F path
```

`live_measurement_enabled()` reads `EXEC_LIVE_MEASUREMENT=false` by default (opt-in).

`market_intel_engine/observers/liquidity_estimator.py` gains an optional live-spread override:

```
if EXEC_LIVE_SPREAD_OVERRIDE=true and recent fills exist:
    spread_pips = execution_quality.spread_pips_mean
```

This bridges Phase G's proxy liquidity band with Phase H's measured spread — the observers benefit from the live venue without a rewrite.

## 11. Risk monitoring

`risk_monitor.py` runs on every fill + every position update:

| Guard | Env | Default |
|---|---|---|
| Max concurrent positions | `RISK_MAX_POSITIONS` | 10 |
| Max exposure per pair (base ccy) | `RISK_MAX_EXPOSURE_PAIR` | 100000 |
| Max total exposure (base ccy) | `RISK_MAX_EXPOSURE_TOTAL` | 500000 |
| Daily realised loss cap (pct) | `RISK_DAILY_LOSS_PCT` | 3.0 |
| Rolling 24h realised loss cap (pct) | `RISK_24H_LOSS_PCT` | 5.0 |
| Broker health floor | `RISK_BROKER_HEALTH_MIN` | 0.30 |

Any breach:

1. Writes an `outcome_events` row (`risk_breach` decision type + `severity`).
2. Sets `BROKER_KILL_SWITCH=true` in-memory (persistent across a request but not a restart — operator must acknowledge to clear).
3. Cancels every working order on the broker.
4. Does **not** flatten open positions automatically (operator confirmation required — this is a design deliberate choice; see Q3).

## 12. Closed-learning feedback

`attribution.py` runs on a 5-min orchestrator task and:

1. Fetches every `brain_decision` outcome event without a matching `execution_realised` twin.
2. Joins to `order_requests` (via `brain_decision_id`), to `fill_events`, and to the closing fills that produced realised PnL.
3. Writes one `execution_attribution` row per closed round-trip.
4. Emits an `execution_realised` outcome event tagged with:
   * `strategy_hash`
   * `brain_decision_id`
   * `predicted_score`
   * `realised_pnl`
   * `slippage_pips`
   * `realised_execution_score` (from the six-component composite)
   * `delta_predicted_realised` = `realised_execution_score − predicted_score`

Phase I's meta-learner will consume `delta_predicted_realised` as the training signal for adaptive `BRAIN_W_*` weights — no code changes needed there because the outcome-events schema is stable.

## 14. Explainability

Every autonomous execution decision emits an `outcome_events` row:

| decision_type | Emitted by | Metrics | Evidence |
|---|---|---|---|
| `order_state_change` | order_lifecycle on every transition | `request_id`, `from`, `to` | `broker_reason`, `qty_filled_so_far` |
| `position_closed` | position_lifecycle on CLOSE | `position_id`, `realised_pnl`, `strategy_hash` | round-trip trace |
| `broker_health_check` | broker_health task | `score`, `band` | rolling components |
| `execution_quality_refresh` | quality snapshot upsert | pair, `score`, `method` | per-component means |
| `execution_realised` | attribution.close | `strategy_hash`, `predicted_score`, `realised_execution_score`, `delta_predicted_realised` | fill_ids, brain_decision_id |
| `risk_breach` | risk_monitor | guard name, severity | before/after snapshot |
| `broker_kill_switch` | manual or automatic | operator, reason | working_orders_cancelled |

Chained query "**Why did strategy X lose money on this trade?**":

1. `execution_realised` → shows realised_pnl vs prediction.
2. `position_closed` → shows entry / exit prices.
3. `order_state_change` → shows every hop in the order lifecycle.
4. `brain_decision` (Phase F) → the original decision.
5. `market_intelligence_refresh` (Phase G) → the market context.
6. `structural_change_detected` (Phase G) → any market shift active at the time.

Full traceability from a broker fill back to a raw price observation.

## 16. Feature flags & rollback

* `EXEC_ENABLED=false` → completely dormant (registry sees no adapter, task list skips execution tasks, endpoints return 503 with a descriptive body). **Instant rollback.**
* `BROKER=paper` → live venue disabled without touching any code. **Deterministic replay for CI / debugging.**
* `BRAIN_USES_LIVE_EXECUTION=false` → Phase F estimator is still the brain's execution-quality input. **Zero behaviour change.**
* `BROKER_KILL_SWITCH=true` → all new submits blocked, working orders cancelled, but the ledger keeps reading — safe emergency mode.
* `PORTFOLIO_POLICY=phase_d` (Phase F rollback) still routes around the brain entirely.

Every knob is a one-line env change — no code deploy required.

## 19. VPS deployment considerations

* **Persistent websocket** — cTrader OAuth session is long-lived. The FastAPI worker owns the socket; if the process restarts, `broker_session_refresh` reconnects on boot from the stored refresh token. **Do not run multiple FastAPI workers with the same account id** — a leader-election check in `broker/ctrader/session.py` acquires a Mongo advisory lock (collection `broker_leader_lock`) and downgrades the loser to read-only paper mode. Operator-visible via `/api/execution/broker/health`.
* **Firewall / egress** — cTrader Open API uses `demo.ctraderapi.com:5035` and `live.ctraderapi.com:5035` (Protobuf/TLS). The `infra/compose/docker-compose.prod.yml` egress rules must permit these hosts.
* **Secrets** — `CTRADER_CLIENT_SECRET` MUST be provided via `.env` (never committed). Mongo-persisted OAuth refresh tokens are AES-encrypted at rest with `EXEC_SECRETS_KEY` (32-byte hex, generated via `openssl rand -hex 32`).
* **Clock** — Slippage / latency measurements rely on host clock accuracy. Add `chronyd` (or verify systemd-timesyncd) to the VPS boot recipe. Any observed drift > 200ms writes a `clock_drift` outcome event and increases broker health penalty.
* **Disk** — `fill_events` + `order_requests` + `execution_attribution` are permanent (audit trail). Estimate ~150 KB/day per active pair at 100 fills/day. VPS disk provisioning MUST factor 5+ years retention.
* **Backup** — Mongo backup cadence for execution collections is now business-critical. Add nightly `mongodump` to `infra/scripts/backup.sh` for those 6 collections specifically.
* **Rollback drill** — Document the exact one-line env change to disable Phase H (`EXEC_ENABLED=false && sudo supervisorctl restart backend`) in `infra/RUNBOOK.md`.
