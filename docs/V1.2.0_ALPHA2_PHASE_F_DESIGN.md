# Phase F — Adaptive Trading Brain

**Status:** DESIGN ONLY (per operator instruction — architecture review required before implementation).

---

## 1. Objective

Evolve the Master Bot from a **tier-based portfolio selector** (Phase C+D output — "here are the top 30 strategies, split into Tier 1/2/3, activate one for the current regime") into a **continuously-adapting portfolio manager brain** that:

* Evaluates every strategy every tick against a rich signal set
* Anticipates regime transitions before they complete (pre-positioning)
* Decides per-strategy actions along a **continuous action space** (weight ∈ [0..1]) rather than the discrete ACTIVATE/PAUSE/REDUCE of Phase D
* Learns which of its own past decisions were correct, and updates its decision policy accordingly

Phase D added the **mechanics** of autonomous rebalancing. Phase F adds the **cognition** — a scoring + policy layer that consumes far more context than Phase D's allocation engine and produces smoother, faster, and (over time) better-quality decisions.

---

## 2. What Phase F is NOT

* NOT a replacement for Phase D. Phase D's engines (allocation, capital, health, promotion, retirement, closed-learning) remain in place — the Adaptive Trading Brain sits **above** them as a policy layer.
* NOT an ML training system. It is a deterministic rule-based scorer with configurable weights that are **updated** by closed-learning feedback — no gradient descent, no neural nets in this phase.
* NOT a new orchestrator task category. It plugs into the existing `self_rebuild` task via a new pluggable policy strategy.

## 4. New modules

Under `engines/brain/` (parallel to `engines/intelligence/` and `engines/portfolio/`):

| File | Responsibility |
|---|---|
| `signals.py` | `SignalCollector` — pulls every input and caches per-tick with TTL |
| `regime_transition.py` | `RegimeTransitionDetector` — compares recent-window regime vs longer-window regime, computes transition probability (0..1) using stability/drift heuristics |
| `execution_quality.py` | `ExecutionQualityScorer` — reads live PnL vs backtest expectancy, produces 0..1 score of "did the strategy actually deliver what the backtest promised?" |
| `risk_budget.py` | `RiskBudget` — tracks running-risk consumption (open positions × per-trade-risk × correlation-adjusted VaR proxy); emits headroom fraction 0..1 |
| `scorer.py` | `StrategyScorer` — pure function producing `score_now`, `score_next` |
| `policy.py` | `DecisionPolicy` — converts scores + risk envelope to actions + target weights |
| `brain.py` | `TradingBrain` — orchestrates signals → scorer → policy → emits `BrainDecision`s |
| `types.py` | Dataclasses: `BrainSignals`, `StrategyScore`, `BrainDecision`, `BrainReport` |

### Public API
```python
from engines.brain import TradingBrain, BrainReport

report = await TradingBrain().tick(state, regime, prices=None)
# report.decisions: List[BrainDecision]  — one per strategy
# report.pre_staged: List[str]           — hashes preparing for next regime

```

## 6. Scoring function (deterministic, transparent)

```python
score_now(s, sig) =
        0.20 * regime_fit(s.style, sig.regime)
      + 0.20 * s.classification_confidence
      + 0.15 * s.recent_pf_normalised          # rolling
      + 0.10 * s.long_pf_normalised            # backtest
      + 0.10 * (1 − s.recent_dd_normalised)
      + 0.08 * s.prediction_accuracy_rolling   # closed learning
      + 0.07 * (1 − pairwise_corr(s, portfolio))
      + 0.05 * session_fit(s.style, sig.session)
      + 0.05 * liquidity_fit(s.style, sig.liquidity_band)
```

```python
score_next(s, sig) =
    if sig.transition_probability >= 0.5 and sig.predicted_next_regime:
        replace sig.regime with sig.predicted_next_regime in score_now
    else:
        0.5 * score_now(s, sig)   # if no transition expected, next ≈ now
```

Weights are **env-driven** so operator can retune without code changes:
`BRAIN_W_REGIME_FIT`, `BRAIN_W_CONFIDENCE`, `BRAIN_W_RECENT_PF`, …

**Every weight is a first-class explainability field** in `StrategyScore.components`, so the operator can inspect any decision and see exactly which signal drove it.

## 7. Policy — from score to action

```python
def decide(score, s, sig, portfolio) -> BrainDecision:
    if score.score_now >= 0.75:
        return TRADE_NOW / INCREASE  (target_weight = clip(0.15..0.25))

    if score.score_now < 0.25 or dd exceeds retire threshold:
        return RETIRE     (if extreme) or REPLACE

    if score.score_now < 0.4:
        return PAUSE      (target_weight = 0.0)

    if 0.4 ≤ score.score_now < 0.75:
        return TRADE_NOW  (target_weight ∝ score_now, capped by style share)

    # Pre-staging (only if transition_probability ≥ 0.5)
    if score.score_next ≥ 0.75 AND s.status == "paused":
        return PRE_STAGE  (small allocation seed, e.g. 0.02–0.05)
```

Every decision is bounded by `PORTFOLIO_MAX_STYLE_SHARE` and the risk-budget headroom from `RiskBudget`.

## 8. Regime Transition Detector

Cheap heuristic (no ML):
* Compute regime on last-N bars (short window, e.g. 30)
* Compute regime on last-3N bars (medium window)
* If short ≠ medium → transition in progress, probability = 1 − (agreement_ratio)
* `predicted_next_regime` = short-window classification

Emits: `{current, medium, predicted_next, transition_probability, evidence}`.

More sophisticated Bayesian / HMM approaches deferred to a later phase.

---

## 9. Risk Budget

Tracks (in memory, seeded from Mongo `open_positions` if present):
* `open_positions_count` — active trades
* `risk_per_trade_pct` (from strategy risk profile)
* `avg_correlation` — reduces effective risk headroom when high
* `daily_realised_pnl` (optional, from execution feed)

Emits `headroom` ∈ [0..1]:
```python
theoretical_max_open = MAX_CONCURRENT_TRADES  (env, default 6)
in_use               = open_positions_count + correlation_penalty
headroom             = 1 - in_use / theoretical_max_open
```

Below `headroom < 0.2` the brain blocks INCREASE/TRADE_NOW and forces REDUCE on the lowest-scoring active strategy.

## 10. Integration with Phase D

`rebuilder.rebuild_master_bot` gets an env-driven policy switch:

```python
if os.environ.get("PORTFOLIO_POLICY", "phase_d") == "brain":
    brain_report = await TradingBrain().tick(state, regime)
    actions = [_brain_decision_to_portfolio_action(d)
               for d in brain_report.decisions]
    # rest of the rebuild pipeline is unchanged
else:
    actions = allocation_decisions(state, regime=regime)   # Phase D default
```

This is strictly additive:
* When `PORTFOLIO_POLICY` is unset or `phase_d`, behaviour is identical to today.
* When `PORTFOLIO_POLICY=brain`, the brain replaces the Phase D allocation engine — everything else (capital, health, promotion, retirement, closed learning, selector, outcome events) is reused unchanged.
* Rollback is a one-env-var flip.

## 15. Open architecture questions for operator review

Before I implement, please confirm:

* **Q1 · Continuous weight action space.** Phase D uses discrete `INCREASE +3%` / `REDUCE -2%`. Phase F design proposes **continuous target_weight ∈ [0..1]** clamped by style-cap. This produces smoother allocations but larger churn potential. Do you want a hard maximum-delta-per-tick cap (e.g. no strategy weight can move more than ±5% per rebuild) to guarantee production stability? Default suggestion: yes, `BRAIN_MAX_WEIGHT_DELTA_PER_TICK=0.05`.
* **Q2 · Pre-staging safety.** When `transition_probability ≥ 0.5` and a paused strategy scores high on `score_next`, the brain proposes `PRE_STAGE` with a small seed weight (0.02–0.05). Should pre-staged strategies actually receive capital (real risk) or should this be a "shadow" allocation that gets promoted to real capital only when the transition completes? I recommend shadow first, real second — but this is a policy call.
* **Q3 · Regime Transition Detector.** The design uses a simple short-window vs medium-window comparison. Are you OK with this heuristic-only start, or should we use an HMM / change-point detector from day one? Heuristic-only lets us ship Phase F fast; HMM adds ~500 LOC of new statistics.
* **Q4 · Execution quality.** In this pod there is no live PnL feed. Should `ExecutionQualityScorer` degrade to `1.0` (assume "matches backtest") when no live feed is present, or be omitted from the score entirely until a real execution feed exists? Default suggestion: return 0.8 when no feed (small negative bias vs assumed-perfect) so the score reflects the missing data.
* **Q5 · Policy learning.** Phase F's weights are env-driven. Closed learning currently only feeds back to `prediction_accuracy_rolling` at the strategy level. Should we also close the loop on the *brain's own weights* — e.g. after N ticks, if the brain-selected active strategy consistently underperforms the second-best candidate, does the brain automatically de-weight `regime_fit` in favour of `recent_pf`? I recommend deferring this "meta-learning" step to Phase G.

## 17. Success criteria for Phase F sign-off

* `PORTFOLIO_POLICY=phase_d` behaviour byte-identical to today (regression test).
* `PORTFOLIO_POLICY=brain` produces smoother allocations (measured: |Δweight| per rebuild drops vs Phase D by ≥ 30% on the same strategy set).
* Brain emits `brain_decision` + `brain_tick` events with full `components` breakdown for every strategy on every tick.
* Pre-staging works: with `transition_probability ≥ 0.5`, a previously-paused strategy that fits the predicted next regime receives a small `target_weight` before the transition completes.
* Risk-budget headroom is respected: with `headroom < 0.2`, no INCREASE/TRADE_NOW is dispatched.
