# Phase B.2 — Unified Autonomous Orchestration Engine

**Status:** Design · to be implemented incrementally with regression tests at each milestone.
**Additive · backward compatible · strict rollback safety.**

---

## 1. Vision

Replace the collection of independent schedulers (`auto_scheduler`, `orchestrator_scheduler`, `bi5_realism_sweep_scheduler`, `auto_data_maintainer`, `continuous_scheduler`) with a **single central decision engine** that answers, every tick:

> *"Given the current system state, available resources, dependencies, AI budgets, and business objectives, what is the highest-value work that should execute next?"*

The end-state factory workflow is fully autonomous:

```
Market Data → Knowledge Refresh → Strategy Generation → Backtesting →
Validation → Mutation → Optimization → Learning → Ranking →
Master Bot Bundle Refresh → (operator review) → deploy
```

The operator's role shrinks to reviewing recommended Master Bot bundles and approving production deployment.

---

## 2. Architectural principles

1. **Additive · backward compatible.** Existing schedulers are NOT deleted. Each gains a `subordinate_to_orchestrator` flag (already the pattern used by `engines/auto_scheduler.py`). When the orchestrator is enabled, subordinate schedulers become dormant. When disabled, everything runs exactly as before.
2. **Extensible task registry.** New task types can be added without touching orchestrator internals — they just implement a `Task` protocol and register.
3. **Signal-driven, not timer-driven.** Fixed intervals disappear except as freshness SLAs (upper bound on how stale a task's output may go). All prioritisation is driven by state signals.
4. **Deterministic priority function.** Same inputs → same score. No randomisation.
5. **Fail-open.** Every component gracefully degrades to conservative behaviour when a signal (probe, budget tracker, dependency reader) is unavailable.
6. **Read-only signals.** The orchestrator NEVER mutates task business state directly — it invokes existing engines through thin adapters.
7. **One architectural change per phase.** Runtime optimisations (process pool, Mongo batching, Prometheus, `_SIG_LOCK` refactor) are explicitly out of scope for this phase.

## 4. Task protocol

Every task is a small class in `engines/orchestrator/tasks/*.py` that satisfies:

```python
class Task(Protocol):
    NAME:                   str          # unique key ("learning_cycle", …)
    WORKLOAD_CLASS:         WorkloadClass
    DEPENDS_ON:             tuple[str, ...]  # NAMEs of prerequisites
    MIN_INTERVAL_S:         int              # freshness SLA (0 = event-driven only)
    PRIORITY_BASE:          float            # 0..100 static baseline
    CPU_ESTIMATE_CORES:     float
    RAM_ESTIMATE_MB:        int
    EXPECTED_DURATION_S:    float
    AI_PROVIDER_REQUIRED:   bool
    COST_ESTIMATE_USD:      float            # per invocation (0 for local)
    BUSINESS_VALUE:         float            # 0..1

    async def readiness(ctx: OrchestratorContext) -> Readiness:
        """Return {"eligible": bool, "reason": str, "pressure": float}.
        `pressure` is a 0..1 multiplier reflecting how overdue this task is
        (e.g. staleness relative to MIN_INTERVAL_S)."""

    async def run(ctx: OrchestratorContext) -> TaskResult:
        """Execute one invocation. MUST be idempotent w.r.t. underlying
        engine (i.e. crashing mid-run must not corrupt state)."""
```

`OrchestratorContext` carries the current adaptive-concurrency snapshot, budget tracker handle, seed defaults, and a run_id.

## 5. Priority scoring function

Deterministic, pure:

```
score = PRIORITY_BASE
      × BUSINESS_VALUE
      × pressure_multiplier       # from readiness() — staleness / dep_debt
      × dependency_readiness      # 0..1 — fraction of deps satisfied
      × budget_headroom_factor    # 1.0 if budget available, tapers to 0
      ÷ resource_cost_factor      # CPU + RAM + provider load penalty
```

`resource_cost_factor` is normalised so tasks that would OOM the box (RAM > available headroom × 0.7) or exceed `adaptive_concurrency.max_concurrent_<class>` are automatically deprioritised toward 0.

Ties broken by:
1. Task with the older `last_completed_ts` first.
2. Otherwise alphabetical `NAME` for determinism.

## 6. Budget tracker (new)

`engines/orchestrator/budget_tracker.py` — pure in-memory rolling-window state with periodic Mongo persistence for restart survival.

Tracks:

| Layer | Data                                                  |
|-------|--------------------------------------------------------|
| Provider RPM window | rolling 60 s deque per provider              |
| Provider daily $    | running sum since UTC midnight per provider  |
| Global daily $      | sum across all providers                     |
| Global monthly $    | sum since UTC 1st                            |

Public API:

```python
class BudgetTracker:
    def can_afford(provider: str, est_cost_usd: float) -> tuple[bool, str]: ...
    def record(provider: str, cost_usd: float, tokens: int) -> None: ...
    def snapshot() -> dict: ...
    def choose_provider(candidates: list[str],
                        est_cost_usd: float,
                        weights: BudgetWeights) -> str | None: ...
```

`choose_provider()` implements the cost/quality/latency/availability weighted pick per Q4.

Env-configurable ceilings:

| Var | Default | Purpose |
|---|---|---|
| `ORCH_BUDGET_DAILY_USD_GLOBAL` | `50.0` | Hard daily USD cap across all providers |
| `ORCH_BUDGET_MONTHLY_USD_GLOBAL` | `1000.0` | Hard monthly USD cap |
| `ORCH_BUDGET_DAILY_USD_<PROVIDER>` | unset | Per-provider daily cap; unset = no per-provider cap |
| `ORCH_BUDGET_RPM_<PROVIDER>` | `60` | Provider RPM cap |
| `ORCH_BUDGET_WEIGHT_COST` | `0.4` | Weight on cost when picking provider |
| `ORCH_BUDGET_WEIGHT_QUALITY` | `0.4` | Weight on quality score |
| `ORCH_BUDGET_WEIGHT_LATENCY` | `0.15` | Weight on median latency |
| `ORCH_BUDGET_WEIGHT_AVAILABILITY` | `0.05` | Weight on circuit-breaker state |

| 10| `ranking`               | API_HOT  | `backtest` (≥10 since last) | 900 s         | 50            | 0.8            |
| 11| `master_bot_bundle_refresh`| API_HOT| `ranking`                   | 1800 s        | 45            | 0.85           |

Every adapter is a thin wrapper. Existing engines (`engines.strategy_engine`, `engines.backtest_engine`, `engines.mutation_engine`, `engines.optimization_engine`, `engines.strategy_ranking_engine`, `engines.master_bot_engine`, `engines.knowledge`, `engines.bi5_realism`, `engines.data_engine.auto_data_maintainer`, `engines.learning.supervisor`) are the source of truth. The orchestrator adds NO business logic — it decides *when* and *whether* to invoke them.

**"Passive mode" (Q1 direction):** an adapter may declare `PASSIVE = True` on first release, which makes `readiness()` always return `eligible=False, reason="passive"`. The task appears in `/api/orchestrator/tasks` but is never dispatched — allowing scaffolding to ship without runtime effect until the operator activates it.

Initial passive set (until stability is proven): `optimization`, `master_bot_bundle_refresh`. Active on day one: everything else.

## 8. Subordinate mode for legacy schedulers

Every existing scheduler grows:

```python
async def _is_subordinated() -> bool:
    if not cfg.get("subordinate_to_orchestrator", True):
        return False
    try:
        from engines.orchestrator import is_active
        return is_active()
    except Exception:
        return False
```

When `_is_subordinated()` returns True, the scheduler tick short-circuits and increments a `subordinate_skip_count` counter (already the pattern used by `auto_scheduler`). The scheduler's own APScheduler cron continues to fire on schedule — it just doesn't do work while the orchestrator is authoritative. This gives us instant rollback: `POST /api/orchestrator/stop` and every legacy scheduler resumes.

## 9. Public API surface

Additive under `/api/orchestrator/*`:

| Endpoint | Method | Description |
|---|---|---|
| `/api/orchestrator/start` | POST | Enable orchestrator (admin) |
| `/api/orchestrator/stop` | POST | Disable orchestrator (admin) |
| `/api/orchestrator/status` | GET | Runtime snapshot: running, tick_count, in-flight tasks, budget state, last-tick derivation |
| `/api/orchestrator/tasks` | GET | List every registered task + its metadata + last_completed_ts + eligibility |
| `/api/orchestrator/tasks/{name}/dispatch` | POST | Manual one-shot dispatch for testing (admin) |
| `/api/orchestrator/tasks/{name}/passive` | POST | Toggle passive mode per task (admin) |
| `/api/orchestrator/budget` | GET | Full budget snapshot |
| `/api/orchestrator/decisions` | GET | Last 100 tick decisions (which tasks were considered / scored / dispatched / rejected + reason) |

All admin-gated except `/status`, `/tasks`, `/budget`, `/decisions` which are read-only for authenticated users.

## 13. Rollback path

At any point in production:

```bash
# runtime toggle — no restart, no rebuild
curl -X POST -H "Authorization: Bearer $TOKEN" \
  http://backend:8001/api/orchestrator/stop
# every legacy scheduler resumes on its own APScheduler cron within its own tick.
```

Or via env:
```
ORCHESTRATOR_ENABLED=false → next boot, orchestrator dormant; legacy schedulers full authority.
```
