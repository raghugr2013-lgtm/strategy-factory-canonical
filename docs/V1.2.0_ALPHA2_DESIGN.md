# Strategy Factory v1.2.0-alpha2 — Learning Milestone Design

**Milestone objective.** Convert Strategy Factory from a stateless generator into a **self-improving AI Strategy Engineering Platform** — every operation writes structured learning data, every future generation is informed by the accumulated outcome ledger, and every artefact carries its full genealogy.

**Delivered as one coherent milestone**, not a series of unrelated patches. Individual PRs land under a shared `v1.2.0-alpha2/*` branch prefix and every commit message references this doc.

---

## 1. Structural review of the current architecture

Assessed with the milestone objective in mind. Findings:

| # | Finding | Impact | Recommendation for v1.2.0-alpha2 |
|---|---|---|---|
| 1 | **Two-router surface** (`app/*` Phase-1 core + `legacy/api/*`) — routing already fixed in `81cd724`, but ownership boundaries are fuzzy. | Medium | Learning routes live under `legacy/api/learning.py` + `legacy/api/ai_workforce.py`; core stays untouched. Additive, no cross-tree refactor. |
| 2 | **LLM calls have per-call log rows** (`llm_call_log`), but no *rollup* per (provider, task, day) and no circuit state. | High — blocks provider scoring | New `llm_call_stats` view collection + in-process circuit-breaker state; existing `log_llm_call` unchanged. |
| 3 | **Strategy documents carry no lineage.** `generate/mutate/repair` write to `strategies` and `strategy_library` but never link back to the request that produced them. | High — blocks P3 genealogy | Add `lineage` sub-document on every stored strategy: `{learning_run_id, parent_hash, provider, model, prompt_version, retrieval_context_hash, token_usage, generation_ms, estimated_cost_usd, stage_chain[]}`. Backward-compatible (readers never expect it). |
| 4 | **Six pipeline endpoints exist in isolation** (generate, validate, repair, backtest, optimize, mutate). No shared correlation ID; retries reset the chain. | High — blocks P1 and P3 | Central `learning_run_id` (UUID) generated at the *first* stage and propagated via request header `X-Learning-Run-Id` OR body field. `emit_outcome` decorator around each endpoint writes an `outcome_events` row. |
| 5 | **Knowledge indexer folds only from strategy_library** (metadata retrieval). | Medium | Extended in this milestone: indexer also folds `outcome_events` aggregates per hash → `strategy_knowledge_index[hash].outcomes`. Retriever score becomes outcome-conditioned (+2.0 for high `validate.pass_rate` + high `operator.avg_rating`). |
| 6 | **VIE client crashes on provider outage** (returns 503 upstream). Circuit-breaker fix from `fb1f675` covers `/api/admin/providers` only, not `run_chat`. | High — production reliability | Wrap `run_chat` in the new `ai_workforce.router.route_call()` that consults circuit-breaker state and rotates through the task-preference chain until a `closed` provider succeeds. |
| 7 | **Operator dashboard is sparse** (Mission Briefing tile only). | Low, but high visibility | New `/api/dashboard/health-mosaic` composite endpoint (P5). Frontend `MosaicRail` component. Ships as the *last* leaf of the milestone once P1–P4 write the data it displays. |
| 8 | **Portfolio intelligence router mounts but is not consulted during generation.** | Medium | Add `portfolio_block` builder in `engines/knowledge/prompt_block.py`, injected *above* the existing `## Prior knowledge` block. Same on-off toggle. |

**Non-recommendations** (repeated from the previous roadmap for transparency):
- No vector DB in this milestone. Metadata retrieval + outcome-conditioning is the proven ceiling test.
- No prompt template rewrite. Additive blocks only.
- No auto-promotion to live trading. Every loop-generated strategy stays behind the manual approval gate.

## 3. Data model — the new collections

### 3.1 `outcome_events` (append-only ledger)
```
{
  _id: ObjectId,
  learning_run_id: UUID,          # correlates every stage in one run
  strategy_hash: str,             # may be null for the initial generate call
  parent_hash: str?,              # set for mutate/repair events
  stage: "generate"|"validate"|"repair"|"backtest"|"optimize"|
         "mutate"|"forward_test"|"approve"|"reject",
  status: "pass"|"fail"|"partial"|"skipped",
  reason: str,                    # short human-readable diagnosis
  metrics: {                      # stage-specific — free-form nested doc
    pf?, dd?, sharpe?, calmar?, trades?, drawdown_days?,
    validation_errors?[], repair_iterations?, optimization_uplift?
  },
  provider: str?,                 # LLM provider if applicable
  model: str?,                    # LLM model if applicable
  prompt_version: str?,           # git-hash or semver of the system prompt
  retrieval_context_hash: str?,   # sha1 over the knowledge_block that was injected
  token_usage: {prompt, completion, total}?,
  duration_ms: int,
  cost_usd: float?,
  operator: {rating: 1..5, comment: str}?,   # for approve/reject
  ts: datetime,
  __v: 1                          # schema version — enables cheap migrations
}
```
Indexes: `(learning_run_id, ts)`, `(strategy_hash, stage)`, `(stage, ts)`, `(provider, ts)`.

### 3.2 `llm_call_stats` (rolling telemetry)
Rolled up asynchronously from `llm_call_log`. Per-provider per-hour bucket:
```
{
  _id: "{provider}:{hour_bucket_iso}",
  provider, model, hour_bucket,
  calls, ok, fail_timeout, fail_network, fail_provider,
  latency_ms_p50, latency_ms_p95,
  tokens_prompt, tokens_completion, tokens_total,
  cost_usd,
  updated_at
}
```
Powers `/api/ai-workforce/health` and the dashboard mosaic tile.

### 3.3 `strategies.lineage` and `strategy_library.lineage` (sub-doc, additive)
```
lineage: {
  learning_run_id, parent_hash?, root_hash?,
  provider, model, prompt_version, retrieval_context_hash?,
  token_usage: {prompt, completion, total},
  generation_ms, estimated_cost_usd,
  stage_chain: ["generate", "validate", "repair", "backtest", ...],  # append per stage
  first_seen_at, last_touched_at
}
```
Every existing read path ignores the new field (Mongo tolerates it). Frontend surfaces it in the strategy-detail drawer.

## 4. Emitter contract (single source of truth for events)

New module `backend/legacy/engines/learning/emitter.py`:

```python
async def emit(
    stage: str,
    *,
    learning_run_id: str,
    strategy_hash: Optional[str],
    status: str,
    reason: str = "",
    metrics: Optional[dict] = None,
    provider: Optional[str] = None,
    model: Optional[str] = None,
    prompt_version: Optional[str] = None,
    retrieval_context_hash: Optional[str] = None,
    token_usage: Optional[dict] = None,
    duration_ms: int = 0,
    cost_usd: Optional[float] = None,
    operator: Optional[dict] = None,
    parent_hash: Optional[str] = None,
) -> str:
    """Write one outcome_events row. Returns the inserted _id (str).
    Never raises — failures are logged and swallowed so pipeline
    correctness is preserved."""
```

Convenience helpers:
- `emit_generate(run_id, hash, provider, ...)`
- `emit_validate(run_id, hash, pass_, errors, ...)`
- `emit_repair(...)`, `emit_backtest(...)`, `emit_optimize(...)`, `emit_mutate(...)`
- `emit_operator_decision(run_id, hash, rating, comment)`

An `@emit_outcome("stage_name")` decorator wraps existing endpoint handlers with zero business-logic changes.

## 5. AI Workforce — provider orchestration

New module `backend/legacy/engines/ai_workforce/`.

- **`circuit_breaker.py`** — in-process `{provider: State}` where `State = closed | half_open | open`, based on rolling 30-call window error rate. Transitions: `closed → open` on ≥ 40 % error rate; `open → half_open` after 60 s; `half_open → closed` on next success or `→ open` on next failure.
- **`telemetry.py`** — every `run_chat` call goes through here. Captures `(provider, model, ok, latency_ms, prompt_tokens, completion_tokens, http_status, error_class, cost_usd)` → in-process ring buffer + async batched insert to `llm_call_stats`.
- **`router.py`** — `route_call(task, prompt, system_message)` iterates the task-preference chain, skipping `open` providers, calling VIE via `run_chat`, and returning the first successful `(text, provider, model)`. If every provider is `open` it returns `(None, None, None)` and the caller falls back to offline generation.
- **`scorer.py`** — per-provider **quality score** derived from outcome_events: how often strategies from this provider survive validate → backtest → optimize. Exposed as `/api/ai-workforce/scores`. Provides a data-driven complement to the static task_preference list.

New endpoints (mounted `/api/ai-workforce/*`):
- `GET /api/ai-workforce/health` — per-provider `{state, latency_p50, error_rate_1h, tokens_1h, cost_1h_usd, last_success_ts, last_error, model}`
- `GET /api/ai-workforce/providers` — configuration + circuit state + effective preference chain
- `GET /api/ai-workforce/scores` — provider quality scores from outcome_events
- `POST /api/ai-workforce/circuit/{provider}/reset` — admin manual reset

## 6. Continuous learning supervisor (P3)

Thin version, shipped in this milestone. Deferred behaviour flagged in code so future depth is additive.

```python
# backend/legacy/engines/learning/supervisor.py
async def run_learning_cycle(seed: LearningSeed) -> LearningRun:
    run_id = uuid4().hex
    await emit_generate(run_id, ..., provider=..., ...)
    strat = await generate_strategy(seed)
    ok = await validate_strategy(strat)
    await emit_validate(run_id, strat.hash, ok, ...)
    if not ok:
        strat = await repair_strategy(strat)
        await emit_repair(run_id, strat.hash, ok=strat.repaired_ok, ...)
    bt = await backtest_strategy(strat)
    await emit_backtest(run_id, strat.hash, bt.summary, ...)
    if bt.pf < 1.2:
        return await _early_reject(run_id, strat)
    opt = await optimize_strategy(strat)
    await emit_optimize(run_id, strat.hash, opt.uplift, ...)
    mut = await maybe_mutate(strat, opt)
    await emit_mutate(run_id, strat.hash, mut.children, ...)
    await knowledge_indexer.incremental_rebuild()
    await strategy_ranker.rerank_hash(strat.hash)
    await stamp_lineage(strat, run_id, ...)
    return LearningRun(id=run_id, terminal_hash=strat.hash, ...)
```

Exposed via:
- `POST /api/learning/cycles` — kick off (admin) with body `{seed_pair, seed_timeframe, seed_style, count}`
- `GET  /api/learning/cycles/{run_id}` — status + event stream
- `GET  /api/learning/lineage/{strategy_hash}` — full genealogy walk (parent_hash chain + all outcome_events)

**Ranker.** Trivial in this milestone: `rerank_hash` recomputes a *learning score* per strategy = weighted sum of best_pf, best_dd, validate.pass_rate, operator.avg_rating, mutation-derived-child-success-rate. Stored back on `strategy_knowledge_index[hash].learning_score` and used by the retriever to bias winners.

## 10. Phase B — SHIPPED (v1.2.0-alpha2-phaseB, 2026-02-15)

Phase B lands the four self-improving-loop pieces on top of Phase A's
ledger. Every one is strictly additive: no legacy module modified, no
route removed, no existing endpoint shape changed. Backend boot log still
reports `legacy full-recovery mount: 92 routers/attachers online`.

### 10.1 Continuous Learning Supervisor
File: `backend/legacy/engines/learning/supervisor.py`

Orchestrates one full cycle under a shared `learning_run_id`:

    generate → backtest (env-gated thresholds) → optimize (placeholder)
      → mutate (placeholder) → rerank (incremental knowledge rebuild)

Every stage emits an outcome_events row and (where applicable) stamps
lineage. Never raises — status is `completed | early_reject | failed`.
Backed by an in-process ring buffer of the last 100 runs for the
dashboard mosaic.

Also ships a **scheduler**: `LEARNING_SCHEDULER_ENABLED=true` auto-starts
a background loop that fires one cycle every `LEARNING_SCHEDULER_INTERVAL_S`
seconds (default 3600). Manual `POST /api/learning/scheduler/{start,stop}`
also exposed for admin control.

### 10.2 Strategy Lineage
File: `backend/legacy/engines/learning/lineage.py`

`stamp_lineage(strategy_hash, learning_run_id, stage, provider, model,
prompt_version, retrieval_context_hash, parent_hash, root_hash,
token_usage, generation_ms, estimated_cost_usd, extra)` merges a
`lineage` sub-document onto every matching doc across `strategies`,
`strategy_library`, `strategy_library_archive`. Uses `$set` + `$addToSet`
for idempotency; `$setOnInsert` writes `first_seen_at` exactly once.

Never raises. Never creates ghost documents — only updates existing rows.

### 10.3 Outcome-conditioned Retrieval
File: `backend/legacy/engines/knowledge/outcome_conditioning.py`

Rolls up `outcome_events` per `strategy_hash` and computes a boost:

    outcome_boost = W * ((0.5 * validate_rate + 0.2 * repair_rate
                          + 0.2 * uplift_norm + 0.1 * approve_rate) - 0.5)

where `W = LEARNING_RETRIEVAL_OUTCOME_WEIGHT` (env, default 2.0).
Applied inside `knowledge.retriever.retrieve()` immediately after the
optional TF-IDF pass. Zero events → zero boost → identical to Phase A
behaviour.

60-second in-process cache; `invalidate_cache()` after every supervisor
cycle keeps the retriever fresh.

### 10.4 AI Workforce Router
Files:
- `backend/legacy/engines/ai_workforce/router.py`  — `route_call(task, ...)`
- `backend/legacy/engines/ai_workforce/scorer.py`  — quality snapshot

Feature-flagged **OFF by default** (`AI_WORKFORCE_FAILOVER=true` to enable).
When enabled:

  1. Resolves the task-preference chain from `AI_WORKFORCE_CHAIN_<TASK>`
     or the global `AI_WORKFORCE_CHAIN` env var (defaults to
     `openai,anthropic,gemini,deepseek,groq,kimi`).
  2. Reorders by cached per-provider quality (`ai_workforce.scorer`).
  3. Skips providers whose in-process circuit-breaker is `open`.
  4. Retries with the next provider on empty output / exception.
  5. Records every attempt to the breaker + telemetry ring.

The public `run_chat` remains binary-compatible: when the flag is off
it takes the legacy VIE path unchanged; when the flag is on it delegates
to `route_call`. Zero recursion — the router uses `_single_attempt`
internally.

### 10.5 Env-configurable thresholds
File: `backend/legacy/engines/learning/config.py`

Every threshold read via `os.environ.get()` at call time (not import
time) so operators can retune without restarting. Every getter is
idempotent and cheap.

| Env var                              | Default | Purpose                                     |
|--------------------------------------|---------|---------------------------------------------|
| `LEARNING_PF_MIN`                    | 1.2     | Early-reject profit factor floor            |
| `LEARNING_DD_MAX_PCT`                | 25.0    | Early-reject max drawdown percentage        |
| `LEARNING_MIN_TRADES`                | 30      | Minimum backtest trade count                |
| `LEARNING_WR_MIN_PCT`                | 25.0    | Minimum win rate                            |
| `LEARNING_WR_MAX_PCT`                | 90.0    | Maximum win rate (overfitting guard)        |
| `LEARNING_OPTIMIZE_UPLIFT_MIN`       | 0.0     | Minimum score uplift to accept optimisation |
| `LEARNING_MUTATION_ENABLED`          | true    | Enable mutation stage                        |
| `LEARNING_SCHEDULER_ENABLED`         | false   | Auto-start periodic scheduler on boot        |
| `LEARNING_SCHEDULER_INTERVAL_S`      | 3600    | Seconds between scheduled cycles             |
| `LEARNING_SCHEDULER_MAX_CONCURRENT`  | 1       | Cap on simultaneous cycles                  |
| `LEARNING_RETRIEVAL_OUTCOME_WEIGHT`  | 2.0     | Scalar boost for outcome-conditioning        |
| `LEARNING_RETRIEVAL_MIN_EVENTS`      | 3       | Min events before conditioning kicks in      |
| `AI_WORKFORCE_FAILOVER`              | false   | Enable router failover (opt-in)              |
| `AI_WORKFORCE_CHAIN[_TASK]`          | (defaults) | Comma-separated provider preference chain |
